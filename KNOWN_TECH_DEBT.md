## Category 1: Intentionally Deferred (Blocked by Design)

### TD-001: Three Duplicate Grid Position Types
**Status:** DEFERRED — requires coordinated refactor
**Files:**
- `aidm/schemas/intents.py` → `GridPoint` (bare x/y, no methods)
- `aidm/schemas/targeting.py` → `GridPoint` (has `distance_to()` with 1-2-1-2 diagonal math)
- `aidm/schemas/attack.py` → `GridPosition` (has `is_adjacent_to()` for AoO checks)

**Why three types exist:** They were built in separate CPs (voice intents, targeting, combat) before a canonical type was planned. Each has different methods needed by its consumers.

**Why NOT to fix now:**
- Unifying them requires updating every import and every test that touches grid coordinates
- The unified type needs `distance_to()`, `is_adjacent_to()`, `to_dict()`, `from_dict()`, and `__eq__()`
- This should be its own CP with dedicated tests, not a drive-by fix

**Correct behavior for agents:** Use the type documented in `AGENT_DEVELOPMENT_GUIDELINES.md` Section 3 for your context (targeting vs AoO vs voice). Do not create a fourth type.

### TD-002: Replay Runner Handles Only 4 of 41+ Event Types
**Status:** DEFERRED — by design
**File:** `aidm/core/replay_runner.py` → `reduce_event()`

**Handled:** `no-op`, `set_entity_field`, `add_entity`, `remove_entity`
**Not handled:** 37+ event types (attack_roll, damage_roll, hp_changed, save_rolled, etc.)

**Why this is intentional:** Combat replay uses **re-execution** (same RNG seed produces same output), NOT event reduction. The replay runner only needs to handle entity management events. Combat events are regenerated by re-running the combat with the same seed.

**Why NOT to fix now:**
- Adding all 37+ event types to reduce_event() would be wasted work
- The re-execution strategy is documented in `AGENT_DEVELOPMENT_GUIDELINES.md` Section 6
- If event replay becomes needed later, it should be a dedicated SKR with design doc

**Correct behavior for agents:** If you add a new entity management event type (add/remove/modify entity fields), add it to reduce_event(). If you add a combat/resolution event type, do NOT add it — it will be replayed via re-execution.

### TD-003: play_loop.py `execute_turn()` is 419 Lines
**Status:** DEFERRED — not blocking current CPs
**File:** `aidm/core/play_loop.py`

**The problem:** `execute_turn()` handles player intents, monster intents, policy evaluation, combat resolution routing, AoO interrupts, and event emission in one 419-line function.

**Why NOT to fix now:**
- All 13 CP-12 integration tests pass against the current structure
- Refactoring requires careful preservation of event ordering and AoO interrupt timing
- The next major consumer (CP-18A spellcasting) will likely force this refactor naturally
- Premature refactoring without a concrete new feature to guide it produces worse abstractions

**Correct behavior for agents:** If you need to add a new intent type to `execute_turn()`, follow the existing pattern (player intent block + monster intent block). Flag if the function exceeds 500 lines after your changes.

**Status:** DEFERRED — safe to fix incrementally, but not required
**Scope:** 14 violations across 6 files

**Violations by file:**
| File | Field | Count |
|------|-------|-------|
| `aidm/core/play_loop.py` | `"team"`, `"defeated"` | 3 |

| `aidm/core/aoo.py` | `"position"` | 1 |
| `aidm/core/targeting_resolver.py` | `"position"` | 1 |

**Why this exists:** The `EF` constants class was created after these modules were written. Existing code was not retroactively updated.

**Correct behavior for agents:** ALL NEW CODE must use `EF.*` constants. If you're already modifying one of these files for other reasons, migrating that file's bare strings to `EF.*` is acceptable as part of the change.
