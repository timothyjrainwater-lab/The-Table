# AD-003: Self-Sufficiency Resolution Policy

**Status:** APPROVED
**Date:** 2026-02-12
**Authority:** PM (Opus) + PO (Thunder)
**Source:** PO concern: "the whole point is that the entire program is supposed to be self-sufficient, self-sustaining, self-answering, self-creating without having to reach outside for anything"

---

## Decision

**The system's self-sufficiency comes from deterministic generation and explicit defaults, not from Spark intelligence.**

When Box needs a fact that doesn't exist in canonical state, the resolution chain is:

1. **Scene canonical data** (SessionBundle, SceneCard) -- authored or pre-generated
2. **Policy Default Library** -- explicit, bounded, deterministic tables for common objects
3. **Seeded Deterministic Generator** -- produces consistent facts from scene_type + seed
4. **Player choice** (last resort) -- constrained options, not free-form

Spark is **never** in this chain. Spark provides language, ideas, narrative texture, and can propose options for the player to choose from. But Spark does not decide mechanical truth.

---

## The Core Tension

The project has two goals that appear to conflict:

1. **Self-sufficient:** The system runs without external intervention. No human DM is needed to fill in world details during play.
2. **Spark cannot supply mechanical truth:** (AD-001) No LLM-generated facts may enter the combat loop.

These goals are compatible when the system's "intelligence" for fact resolution comes from:
- **Authored content** (scene packs, encounter definitions)
- **Deterministic generators** (seeded, reproducible, rules-bounded)
- **Policy defaults** (explicit tables maintained as data, not LLM output)

They are incompatible when "self-sufficient" is interpreted as "the LLM invents whatever is needed." That interpretation violates AD-001 and creates soft nondeterminism.

---

## Spark's Actual Role

Spark is **the intelligence that provides language and ideas**, not "the intelligence that provides information when needed."

| Spark Provides | Spark Does NOT Provide |
|---|---|
| Vivid description of combat | Whether a table provides cover |
| NPC dialogue and personality | NPC stat blocks or ability scores |
| Atmospheric scene texture | Object dimensions, materials, hardness |
| Proposed options for player choice | The mechanical answer to the choice |
| Narration of what happened | What actually happened |

When a player asks "is there anything I can hide behind?", the answer comes from the scene's canonical data or the policy library -- not from Spark. Spark's role is to narrate the result: "You duck behind the heavy oak table."

## The Policy Default Library

### Purpose

Eliminate 80% of NeedFact situations by providing explicit defaults for common game objects. These defaults are:
- **Deterministic** (same object class always produces same properties)
- **D&D 3.5e RAW-grounded** (dimensions and materials produce correct cover/break DC/hardness per DMG)
- **Maintained as data** (not generated by LLM; stored in a versioned registry)
- **Tagged as POLICY_DEFAULT** (provenance is explicit; can be overridden by scene data or player choice)

The library grows over time as new scene types are authored. Priority order:
1. Tavern/inn objects (most common first encounter)
2. Dungeon corridor objects (walls, doors, traps, rubble)
3. Outdoor terrain features (trees, boulders, fences, streams)
4. Temple/church objects (pews, altars, statues)
5. Castle/fortress objects (arrow slits, murder holes, portcullis)

## The Seeded Deterministic Generator

### Purpose

When a scene is loaded but specific objects aren't fully defined, the generator produces complete object facts from:
- **Scene type** (tavern, dungeon_corridor, dungeon_chamber, forest_clearing, etc.)
- **Seed** (deterministic, derived from scene_id: `hash(scene_id) % 2**31`)
- **Policy Default Library** (provides property templates)

### Properties

1. **Deterministic:** Same scene_type + seed = same objects every time
2. **Bounded:** Object counts, dimensions, and placements all come from bounded ranges per scene type
3. **RAW-grounded:** Generated properties map to valid D&D 3.5e values
4. **Replayable:** Output is logged as POLICY_DEFAULT or GENERATED_SEEDED provenance

### Scene Type Templates

Each scene type defines:
- Object class pool (which objects CAN appear)
- Count ranges (min/max per object class)
- Placement rules (against walls, center, corners)
- Lighting default (bright, dim, dark)
- Ceiling height range

Example:
```
tavern_common_room:
  objects:
    tavern_table: {min: 2, max: 6}
    tavern_chair: {min: 4, max: 16}
    bar_counter: {min: 1, max: 1}
    wooden_barrel: {min: 0, max: 3}
    fireplace: {min: 0, max: 1}
  lighting: bright (fireplace + lanterns)
  ceiling_height_ft: {min: 8, max: 12}
  floor_material: wood
```

The seed selects specific values within each range. The generator places objects according to placement rules. All output is committed to the scene's canonical data before play begins.

## Resolution Chain in Practice

### Example: "Is there anything I can hide behind?"

```
1. Box receives: player wants cover
2. Box queries Lens: what objects in current scene are solid and tall enough?
3. Lens checks scene canonical data:
   → Scene has 3 tables, 1 bar counter (from generator or authored content)
   → Table properties from Policy Default: 2.5ft tall, wood, solid
   → Bar counter: 3.5ft tall, wood, solid
4. Box computes: table provides partial cover (+4 AC),
   bar counter provides improved cover (+8 AC)
5. Box presents options via game interface
6. Spark narrates the action: "You dive behind the bar counter
   as arrows splinter the wood above your head."
```

At no point did Spark decide what was in the room or what provided cover.

### Example: "I try to break down the door"

```
1. Box receives: break attempt on door
2. Box queries Lens: what are the door's properties?
3. Lens checks scene canonical data:
   → Door is "wooden_door" class (from scene pack)
   → Properties from Policy Default: hardness 5, HP 15, break DC 15
4. Box resolves: Strength check vs DC 15
5. Result flows through event system
6. Spark narrates: "You shoulder the door with all your might.
   The old wood cracks and splinters under the force."
```

### Example: "What's behind the unexplored door?" (Improvisation)

```
1. Player opens a door that leads to an undefined area
2. Scene pack has: "door_03 → undefined_room"
3. Deterministic Generator activates:
   → Scene type hint: "storage_room" (from scene pack metadata)
   → Seed: hash("dungeon_01_room_04") % 2^31
   → Generates: 8×10ft room, 3 crates, 1 barrel, dim lighting
   → All objects get Policy Default properties
   → All facts committed to canonical state
4. Lens indexes the new room
5. Box can now resolve any mechanical queries about the room
6. Spark narrates the discovery: "The door swings open to reveal
   a cramped storage room, dusty crates stacked along the walls."
```

Spark narrated. The generator created. The distinction is absolute.

## Relationship to AD-001 and AD-002

### AD-001 (Authority Resolution Protocol)
AD-003 specifies WHERE the authoritative facts come from in the NeedFact resolution chain. AD-001 specifies that Spark cannot supply them. AD-003 provides the alternative: Policy Defaults and Deterministic Generation fill the gap that Spark cannot.

The Policy Default Library maps directly to AD-001's Priority Level 2 (policy defaults). The Deterministic Generator is a formalization of what AD-001 calls "explicitly declared rule-backed defaults."

### AD-002 (Lens Context Orchestration)
AD-003 does not affect the Lens→Spark prompt assembly protocol. It affects what data Lens has available to assemble. By ensuring that scene facts are always complete (via generator + defaults), AD-003 eliminates the scenario where Lens must ask Spark for facts to include in the truth channel.
