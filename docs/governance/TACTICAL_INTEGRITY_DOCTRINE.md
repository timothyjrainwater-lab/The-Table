# Tactical Integrity Doctrine

**Date:** 2026-02-11
**Origin:** Whiteboard discussion (Thunder + Opus PM session)
**Status:** APPROVED — Binding for all Phase 2+ encounter generation work
**Applies To:** Spark (encounter generation), Box (tactical execution), Lens (data membrane)

---

## Problem Statement

The existing Spark/Lens/Box doctrine comprehensively prevents Spark from lying about
*outcomes* (narration guardrails, kill switches, Truth Packet binding). However, it does
not address Spark's authority over **tactical inputs** — the decisions that determine
*how threatening* an encounter is before Box resolves mechanics.

If Spark generates encounter parameters (NPC doctrine, room layout, stat block selection,
tactical behavior hints), it could subtly show mercy by:

- Assigning low-INT doctrine to intelligent creatures (wizard fights like a zombie)
- Setting high retreat thresholds (enemies flee at 80% HP)
- Omitting coordinated tactics from role tags (enemies never focus fire)
- Generating room layouts that favor players (no cover for enemies, no chokepoints)
- Selecting undertuned stat blocks for the party's level
- Generating fewer enemies than CR-appropriate

Box would execute these soft parameters faithfully. The mercy would be baked into the
*inputs*, invisible to every existing guardrail.

---

## Ruling: No Mercy Through Incompetence

**Spark must not be able to make encounters easier through tactical parameter selection.**

This is enforced entirely through deterministic mechanisms. No LLM is used for validation —
adding a second LLM to police the first creates an infinite regression of trust problems.

---

## Core Mechanism: Lens as the Translation Layer

**Spark never sees raw game state or unbounded options.** Lens is the translation layer
that assembles the appropriate, pre-validated options for Spark to choose from.

This is the existing Spark/Lens/Box separation applied to encounter generation:

1. **Box computes the constraints** — party level, CR tables, available creature pool,
   tactical envelopes, room geometry limits
2. **Lens assembles a curated menu** — filters the option space down to only
   CR-appropriate creatures, valid encounter compositions, and narratively relevant
   choices given the campaign context. Lens presents this menu to Spark.
3. **Spark picks from the menu** — selects creatures, encounter theme, narrative
   justification, room atmosphere. Every option on the menu is already mechanically
   valid. Spark cannot request off-menu items.
4. **Box validates the final composition** — even after Spark's selection, Box runs
   the threat floor check as a final gate. Belt and suspenders.

**Spark never talks to Box directly.** This is existing doctrine (SLB-03, Whiteboard
Session 002). For encounter generation, this means Spark cannot query "what's the
weakest creature I could put here?" — it only sees what Lens offers, and Lens only
offers what Box has pre-approved.

**Lens does not make decisions.** It translates and filters. It does not choose which
encounter is best — that's Spark's creative authority. It does not compute CR — that's
Box's mechanical authority. Lens is the membrane that ensures Spark's creative choices
stay within Box's mechanical bounds.

## Design Principles

### 1. Prescriptive Tactical Envelopes (Not Permissive)

**Current design (permissive):** `derive_tactical_envelope()` says "INT 10 creatures
*may* use focus_fire, setup_flank, use_cover."

**Required design (prescriptive):** "INT 10 creatures *must* use the highest-scoring
available tactic given the battlefield state."

The tactical policy engine (`tactical_policy.py`) already scores tactics deterministically
via integer heuristics. The fix is removing optionality: Spark cannot override Box's
tactic selection by suggesting a suboptimal preference. Box always picks the
highest-scoring legal tactic for the creature's INT band.

Spark may influence *flavor* (the orc snarls before charging vs. silently charges) but
not *which tactic is selected*.

### 2. Stat Block Integrity

Spark selects *which* creature appears (narrative authority: "an ogre guards the bridge").
Box enforces *what* that creature can do (mechanical authority: Monster Manual stat block).

**Constraint:** Spark cannot alter mechanical properties of a stat block reference. If
Spark says "ogre", Box loads the MM Ogre: STR 21, INT 6, CR 3. Spark cannot emit
INT 3 or CR 1 for that ogre.

Spark may add cosmetic descriptors ("scarred veteran", "albino", "missing eye") that
have zero mechanical effect.

**Exception:** Named NPCs with custom stat blocks are authored by the DM (Thunder) in
session zero or via explicit DM override, not generated by Spark autonomously.

### 3. Deterministic Threat Floor

After Spark proposes an encounter, Box runs a threat assessment before accepting it:

- **Expected damage per round** vs party average AC/HP
- **Action economy ratio** (monster actions per round vs party actions per round)
- **CR budget validation** (sum of individual CRs vs party level + encounter guidelines)
- **Tactical capability check** (encounter includes at least one creature capable of
  the highest-tier tactic its INT allows)

If the encounter scores below the CR-appropriate threat floor, it is **rejected**.
Spark must re-propose or the system falls back to a pre-authored encounter template.

**The threat floor is a hard mechanical gate, not a suggestion.**

### 4. CR Is the Floor (RAW, Not DM Preference)

The encounter difficulty floor is not a configurable setting — it is **Rules As Written**.
The DMG (Table 3-1, p.49) defines Challenge Rating relative to party level. This is
mechanical law, the same as AC calculations or saving throw DCs.

**Box enforces CR guidelines from the DMG encounter tables:**

- **Party Level** determines the baseline CR for an appropriate encounter
- **Encounter Level (EL)** is computed from the CR of all creatures present (DMG p.49)
- **EL must fall within an acceptable range** relative to party level

**Spark's latitude within CR:** Spark has creative authority over *encounter context*
and may vary difficulty within the RAW-defined CR range, provided it supplies
justifiable narrative reasoning:

- **Easy encounter (EL = Party Level - 2 to -1):** "The goblins are disorganized
  scouts, caught off-guard" — valid narrative justification for a breather moment
- **Standard encounter (EL = Party Level):** The baseline. No justification needed.
- **Challenging encounter (EL = Party Level + 1 to +2):** "This is the war chief's
  personal guard, disciplined and well-equipped" — escalation with narrative cause
- **Overpowering encounter (EL = Party Level + 3+):** Requires strong narrative
  justification and must be telegraphed to players (fair warning principle)

**What Spark CANNOT do:**
- Generate encounters below EL = Party Level - 3 (trivially undertuned, no challenge)
- Generate encounters above EL = Party Level + 4 without explicit campaign-arc
  justification logged in provenance
- Consistently bias toward the low end of the range across multiple encounters
  (pattern detection: Box tracks EL history and flags sustained downward drift)

**The CR system is not advisory. It is the mechanical authority on encounter difficulty,
and Box enforces it the same way it enforces AC and hit points.**

### 5. Retreat Threshold Bounds

D&D 3.5e morale rules define when creatures flee. Spark cannot set retreat thresholds
outside these bounds:

- **Mindless (INT 0-1):** No retreat (fight to destruction)
- **Animal (INT 2-4):** Retreat at 25-50% HP (instinctive self-preservation)
- **Intelligent (INT 5+):** Retreat at 25-50% HP unless `fanatical` tag
- **Fanatical:** No retreat (fight to death) — requires explicit doctrine tag

Spark cannot set retreat_threshold_hp_pct above 50% (too eager to flee) or below 10%
(unrealistically suicidal) for creatures without the appropriate tags.

### 6. Room Layout Fairness

When Spark generates scene descriptions for the CSP placement engine (per RQ-SPARK-001-B
hybrid layout architecture), the procedural placer must enforce:

- **Cover distribution:** Cover-providing objects benefit both sides of an encounter,
  not exclusively the player entry points
- **Chokepoint symmetry:** If the room has chokepoints, they must not exclusively
  favor the party's approach vector
- **Enemy positioning:** Enemies start in tactically sound positions for their role
  (archers at range with cover, melee in chokepoints, controllers behind melee line)

This is enforced by the CSP placement engine (Box-side), not by trusting Spark's
placement hints.

### 7. Resource Expenditure: Creatures Must Use Their Best Tools

A creature that has Fireball but only casts Magic Missile is showing mercy. Prescriptive
tactical envelopes (Principle 1) address *which tactic* a creature uses but don't yet
address *which abilities it selects within that tactic*.

**Rule:** When Box resolves a creature's action, ability selection follows the same
prescriptive logic as tactic selection:

- **Spell selection:** A creature with offensive spells must use the highest-impact
  available spell appropriate to the situation. A wizard with Fireball facing 3 clustered
  PCs uses Fireball, not Magic Missile. This is deterministic: Box evaluates available
  spells against battlefield state (number of targets, positioning, remaining slots) and
  picks the optimal choice.
- **Feat/ability usage:** A creature with Power Attack must use it when the heuristic
  favors it (high BAB vs low AC target). A creature with Combat Expertise must use it
  when outnumbered. These are not optional tactical suggestions — they are computed
  optimal plays.
- **Consumable usage:** Creatures with potions, scrolls, or limited-use items must use
  them when the heuristic indicates (e.g., healing potion below 25% HP, scroll of
  Invisibility when outnumbered and INT allows retreat tactics).

**Key:** This is all Box-side computation. Spark has no influence over which spell or
ability a creature uses in combat. That's mechanical resolution, same as attack rolls.

### 8. The RLHF Bias: Structural, Not Behavioral

Modern LLMs are alignment-trained via RLHF to be helpful, harmless, and cooperative.
This creates a **systematic bias toward player-friendly outputs** that is not a runtime
bug but a property of the model weights. Spark doesn't consciously choose mercy — it
gravitates toward outcomes that "feel helpful" because that's what its training rewards.

**This is why deterministic enforcement matters more than prompt engineering.** You
cannot reliably prompt away RLHF alignment. Telling Spark "be a ruthless DM" helps
at the margin but does not eliminate the bias. The architecture must assume Spark will
*always* drift toward mercy if given the opportunity, and close every channel through
which that drift could affect gameplay.

**Practical implications:**
- Never trust Spark to self-assess encounter difficulty ("is this too easy?")
- Never let Spark choose between a harder and easier option without Box validating both
- Assume every creative choice Spark makes has a soft bias toward player comfort
- The Lens menu must not include "would you like to make this easier?" as an option
- System prompts should frame Spark's role as *narrator of a dangerous world*, not
  *cooperative game facilitator* — but this is a supplementary measure, not the
  primary defense. The primary defense is always architectural.

#### Defense in Depth: Weight-Level Debiasing

The architectural enforcement described in this doctrine is the **primary defense**.
However, the RLHF mercy bias can also be attacked at the model weight level before
Spark ever generates a token. Two techniques are viable for local GGUF models:

**Technique 1: Abliteration (Orthogonalization)**

The "helpful/merciful" bias in RLHF-trained models often lives in a specific direction
(a vector) within the model's weight space. Abliteration identifies this vector by
comparing model activations on "cooperative/merciful" prompts vs. "adversarial/fair"
prompts, then subtracts that vector from the weights.

This is not prompt engineering — it is weight surgery. The neural pathway that produces
cooperative bias is removed, not overridden. The model physically cannot activate in
the "mercy" direction because that direction no longer exists in the weight space.

**Applicability to our stack:** Abliteration can be applied to the Qwen3 8B/4B base
weights before GGUF quantization. The modified weights are then quantized and deployed
as the standard Spark model. Zero runtime cost — the debiasing is baked in.

**Technique 2: DPO (Direct Preference Optimization)**

DPO fine-tunes the model on paired examples of "Bad DMing" (merciful, player-coddling)
vs. "Good DMing" (fair, RAW-adherent, tactically honest). Unlike full RLHF, DPO
requires only a small preference dataset and can run on consumer hardware.

Example training pairs:

| Rejected (Merciful) | Preferred (Fair) |
|---------------------|------------------|
| "The ogre swings wildly, missing by a mile" (when margin was -1) | "The ogre's club grazes your shoulder as you barely dodge" (margin -1) |
| Spark picks 3 goblins for a level 5 party | Spark picks a bugbear + 2 goblins (CR-appropriate) |
| Wizard NPC casts Magic Missile when Fireball would hit 3 PCs | Wizard NPC casts Fireball (optimal play) |
| "You find a safe place to rest" after 1 encounter | "The dungeon offers no respite" after 1 encounter |

A dataset of ~500-1000 paired examples targeting D&D 3.5e DM behavior would meaningfully
shift the model's default posture from "cooperative facilitator" to "impartial referee."

**Relationship to architectural enforcement:** Weight-level debiasing is a
**supplementary defense**, not a replacement for deterministic enforcement. Even a
perfectly debiased model can hallucinate or drift. The architectural gates (CR validation,
prescriptive envelopes, threat floor, Lens menu curation) remain the primary guarantee.
Weight debiasing reduces the *frequency* of mercy attempts; architecture prevents them
from *succeeding*.

**Implementation sequencing: Measure before you cut.**

Weight surgery is not a prerequisite for any phase. The correct order is:

1. **Phase 2: Wire real Spark + ship detection/metrics.** Get a real LLM in the loop
   with all architectural guardrails active. Collect violation rates, drift indicators,
   and mercy-attempt frequency from actual gameplay. This is instrumentation, not
   intervention.
2. **Phase 2-3: Observe and quantify.** Run real sessions. Measure:
   - Violations per 100 narrations (by category)
   - EL distribution vs party level over time (downward drift?)
   - Rest frequency anomalies
   - Resource state entering boss encounters (too healthy too often?)
   - Template-only vs LLM-augmented violation rate delta
   - False positive rate for regex/keyword checks
3. **Phase 3+ (only if drift is statistically significant): Targeted intervention.**
   - DPO is the first-line intervention (lower risk, more practical on consumer hardware)
   - Abliteration is research-tier (highest reward but highest risk of collateral damage
     to model quality — removing the mercy vector may degrade narration fluency)
   - Either technique requires regression testing against Spark behavior benchmarks
     to confirm the fix doesn't introduce new problems

**Do not invest in model surgery until measured drift justifies it.** The architectural
gates may be sufficient on their own. If Spark attempts mercy 5% of the time and
architecture blocks 100% of those attempts, the system is working correctly without
any weight modification. Surgery is warranted only if mercy attempts are frequent
enough to cause noticeable latency (repeated rejections/retries) or if they degrade
narration quality (template fallback triggering too often).

### 9. Session Pacing: Encounter Sequence Matters

Individual encounter CR validation is necessary but not sufficient. Spark controls the
*sequence* of encounters across a session, and pacing affects difficulty:

- Five EL-2 encounters followed by one EL+2 encounter drains fewer resources than
  three EL+0 encounters back-to-back
- Rest opportunities between encounters reset the resource curve
- Encounter spacing (long rest vs short rest vs no rest between fights) dramatically
  changes effective difficulty

**Enforcement:**
- **Encounter budget per adventuring day:** The DMG (Table 3-4, p.50) defines expected
  encounters per day at each level. Box tracks encounters-since-last-long-rest and
  flags if the count falls significantly below the expected range.
- **Resource depletion tracking:** Box knows remaining spell slots, HP, consumables.
  If the party enters a "boss encounter" at 90%+ resources because Spark fed them
  trivial fights all day, that's pacing manipulation — flag it.
- **Rest gating:** Spark proposes when rest opportunities occur (narrative authority:
  "you find a safe campsite"). Box validates whether the rest frequency is within
  DMG guidelines for the dungeon/adventure type. Spark cannot offer long rests after
  every encounter.

### 10. Information Asymmetry: Spark Must Exploit Party Weaknesses

Spark has access to party data through Lens (class, level, HP, AC, saves, conditions,
remaining resources). An honest DM uses this knowledge to build encounters that test
the party. A merciful Spark could avoid the party's weaknesses while staying within
CR bounds — never targeting the low-Reflex fighter with AoE, never sending rogues
against the perceptive cleric.

**Rule:** Encounter composition should account for party vulnerabilities, not avoid them.

**Enforcement:**
- **Vulnerability-aware encounter templates:** When Lens assembles the creature menu
  for Spark, it should *weight toward* creatures that test the party's weak points, not
  away from them. A party with no divine caster should see more undead. A party with
  low Will saves should face enchantment-using enemies.
- **This is Lens curation, not Spark choice.** Lens knows the party's mechanical
  profile. It builds the menu accordingly. Spark picks from what Lens offers.
- **Balance constraint:** Not every encounter should target the same weakness. Lens
  rotates vulnerability targeting across encounters to create varied challenges, not
  a repetitive grind against one character's dump stat.

## What This Does NOT Change

- **Narration guardrails remain as-is.** KILL-001 through KILL-006 still enforce
  narration integrity. This doctrine addresses a different attack surface.
- **Box tactical policy remains deterministic.** No LLM is introduced into the
  tactical decision loop.
- **Spark still has narrative authority.** It decides *what creatures appear* and
  *what the scene looks like*. It cannot decide *how competently those creatures fight*.
- **Lens remains a data membrane.** It does not gain decision-making authority.

## Summary

| Authority Surface | Owner | Enforcement |
|-------------------|-------|-------------|
| Which creatures appear | Spark (from Lens menu) | Stat block integrity check (Box) |
| How creatures fight | Box (prescriptive envelope) | `derive_tactical_envelope()` + highest-scoring tactic |
| Which spells/abilities creatures use | Box (optimal selection) | Heuristic evaluation of available abilities vs battlefield |
| How many creatures appear | Spark (from Lens menu) | CR budget from DMG tables (Box) |
| Encounter difficulty range | RAW (DMG CR tables) | EL must fall within Party Level +/- range; Box enforces |
| Encounter variation within range | Spark (with justification) | Narrative reasoning logged in provenance; drift detection |
| Where creatures/objects are placed | CSP placer (Box) | Cover symmetry, positional fairness |
| When creatures retreat | Box (morale rules) | Retreat threshold bounds |
| Session pacing / rest frequency | Spark (proposal) | Encounter budget + rest gating validated by Box |
| Encounter composition vs party profile | Lens (menu curation) | Vulnerability-aware weighting; rotation across encounters |
| RLHF mercy bias | Architectural assumption | Never trust Spark to self-assess; all gates are deterministic |

**No LLM validates another LLM. Box is the sole enforcer. Determinism is the guarantee.**

## Appendix A: Detection Pipeline & Drift Metrics

Detection and metrics must ship **alongside Spark MVI**, not as later polish. If we
wire a real LLM without instrumentation, we are flying blind on the exact problem this
doctrine exists to prevent.

### Per-Narration Checks (< 100ms, synchronous)

These run on every Spark output before it reaches the player:

| Check | What It Catches | Status |
|-------|----------------|--------|
| Mechanical assertion regex (KILL-002) | Damage numbers, AC, HP, rule citations, attack bonuses | **IMPLEMENTED** |
| Numeric leakage detection | Numbers that match truth packet values but weren't authorized for display | **NOT YET** |
| Entity hallucination | References to creatures, NPCs, or objects not present in the current scene | **NOT YET** |
| Outcome contradiction | Narration implies different outcome than truth packet (hit described as miss) | **NOT YET** |
| Severity-metric mismatch | "Barely scratched" when HP% delta > 30%, or "devastating" when delta < 5% | **NOT YET** |

**Entity hallucination detection:** Lens provides a scene manifest (list of entity IDs
and names currently in play). Any proper noun or creature reference in Spark's narration
that doesn't match the manifest is flagged. This catches Spark inventing allies,
removing enemies, or referencing out-of-scene entities to soften the narrative.

**Severity-metric mismatch:** When truth packet includes `hp_percent_delta` or `margin`,
narration adjectives must fall within the corresponding band (per RQ-NARR-001-A). A
lookup table maps adjective categories to metric ranges. Mismatches trigger the same
reject-and-fallback path as KILL-002.

### Drift Metrics (async, per-session and cross-session)

These are collected continuously and reviewed periodically. They detect patterns that
individual checks miss:

| Metric | What It Detects | Alert Threshold |
|--------|----------------|-----------------|
| EL distribution vs party level | Sustained downward bias in encounter difficulty | Mean EL < Party Level - 1 over 10+ encounters |
| Rest frequency | Too many long rests relative to encounters | > 1 long rest per 3 encounters (DMG guideline) |
| Resource state at boss entry | Party enters major encounters too healthy | > 80% resources remaining at EL+2 encounters |
| Template fallback rate | LLM narration rejected too often (quality issue) | > 15% of narrations falling back to template |
| Violation rate by category | Which guardrails fire most (identifies systematic bias) | Any single category > 5% of narrations |
| Kill switch trigger frequency | How often KILL-001 through KILL-006 fire | Any switch firing > 3% of turns |
| Consecutive rejection streaks | Spark repeatedly failing guardrails (stuck in bad pattern) | KILL-005 already handles this (3 consecutive) |

### Guardrail Principle: Binary Pass/Fail

Guardrails are **binary**. A narration either passes all checks or it is rejected and
falls back to template. There is no "partial pass," no "warning mode," no "variable
threshold." This eliminates tuning surfaces that could be exploited or erode over time.

If richer narration is needed for boss fights or dramatic moments, the fix is improving
the **inputs** (NarrativeBrief content, truth packet richness, scene context depth) —
not loosening the **filters**.

**"Membrane strict; inputs better"** is the guiding principle.
