## Executive Summary

This document defines the canonical schema for Scene Fact Packs—structured JSON outputs generated by Spark (LLM layer) and validated by Lens (data membrane) for the D&D 3.5e AI Dungeon Master system. It establishes unit standardization, archetype defaults, unknown-value discipline, and detail drift prevention strategies.

**Key Principles:**
- Spark generates, Lens validates, Box adjudicates
- All Spark output is `[NARRATIVE]` or `[DERIVED]`, never `[BOX]`
- Dimensions in feet, positions in 5ft grid squares, weights in pounds
- Unknown values emit `"UNKNOWN"` literal, never guesses
- First-seen values cached to prevent detail drift

## 1. Scene Fact Pack JSON Schema

### 1.1 Schema Overview

The Scene Fact Pack is a Pydantic-validatable JSON structure with three major sections:
1. **Scene Metadata** - Environmental properties and context
2. **Objects** - Physical entities (furniture, terrain, destructibles)
3. **Creatures** - NPCs and monsters with D&D 3.5e geometry

## 2. Unit Policy

### 2.1 Canonical Units

**All Spark outputs MUST use the following units:**

| Measurement Type | Unit | Notes |
|------------------|------|-------|
| Distance (general) | feet | Never meters, yards, or inches |
| Grid positions | squares | 1 square = 5 feet (D&D 3.5e standard) |
| Elevation/height | feet | Positive = above ground, negative = below |
| Weight | pounds | Never kilograms, ounces, or tons |
| Area effects | radius (feet) | Fireball = 20ft radius, not diameter |
| Speed | feet/round | D&D 3.5e uses feet per 6-second round |
| Temperature | Fahrenheit | Optional, rarely needed |

### 2.2 Conversion Forbidden

Spark **MUST NOT**:
- Convert between metric and imperial
- Mix units (e.g., "5 meters" or "2 yards")
- Use compound units ("3ft 6in" → emit `3.5`)
- Approximate with ranges ("3-4 feet" → emit `UNKNOWN` if uncertain)

### 2.3 Position Coordinates

**Grid Coordinate System:**
- Origin (0,0) at top-left corner
- X-axis increases eastward (right)
- Y-axis increases southward (down)
- All positions are CENTER of occupied square(s)
- Large creatures specify top-left corner of occupied area

**Example:**
```json
{
  "position": {"x": 5, "y": 3},
  "size_category": "Large"
}
// Occupies squares (5,3), (6,3), (5,4), (6,4)
```

### 2.4 Precision Standards

| Value Type | Precision | Justification |
|------------|-----------|---------------|
| Creature positions | Integer squares | D&D 3.5e grid-based |
| Object dimensions | 0.5 ft increments | Reasonable for furniture |
| Elevation | 1 ft increments | Sufficient for gameplay |
| Weight | 1 lb increments | D&D 3.5e standard for <100 lbs |
| Weight (large objects) | 10 lb increments | Reasonable for 100+ lbs |

## 4. Unknown-Handling Protocol

### 4.1 The "UNKNOWN" Discipline

**Core Principle:** Spark MUST emit the string literal `"UNKNOWN"` for any dimension it lacks certainty about. **Never guess, never hallucinate.**

#### When to Emit UNKNOWN

Spark emits `"UNKNOWN"` when:
- Narrative describes object vaguely ("a table" without dimensions)
- Source material lacks specifics (module says "chairs" but not size)
- LLM confidence is low (<70% estimated)
- Regenerating a scene and prior value not accessible

#### When NOT to Emit UNKNOWN

Spark CAN specify values when:
- Narrative explicitly states dimension ("a 10-foot pole")
- Object class has canonical size (e.g., "longsword" is always 3.5ft)
- Source material provides measurement
- Value is plot-critical and was player-specified

### 4.2 Lens Handling of UNKNOWN

```python
# Pseudo-code for Lens validation
def resolve_dimension(object_data, field):
    value = object_data.dimensions.get(field)
    
    if value == "UNKNOWN":
        # Check cache first (prevents drift)
        cached = cache.get(object_data.object_id, field)
        if cached:
            return cached.value, "[CACHED]"
        
        # Use archetype default
        archetype = ARCHETYPE_DEFAULTS.get(object_data.class)
        if archetype:
            default_value = archetype[field]
            cache.set(object_data.object_id, field, default_value)
            return default_value, "[ASSUMED_STANDARD]"
        
        # Fallback
        return GENERIC_DEFAULTS[field], "[FALLBACK_GENERIC]"
    
    else:
        # Spark provided value
        cache.set(object_data.object_id, field, value)
        return value, "[SPARK_GENERATED]"
```

### 4.3 Tagging System

Every resolved dimension receives a provenance tag:

| Tag | Meaning | Authority Level |
|-----|---------|-----------------|
| `[BOX_CANONICAL]` | Box computed from rules | Authoritative |
| `[SPARK_GENERATED]` | Spark provided explicitly | High confidence |
| `[CACHED]` | Retrieved from prior resolution | Stable (prevents drift) |
| `[ASSUMED_STANDARD]` | Archetype default used | Medium confidence |
| `[FALLBACK_GENERIC]` | No archetype, used generic | Low confidence |
| `[PLAYER_SPECIFIED]` | Player directly stated | Overrides all |

### 4.4 Rejection Protocol

Lens **rejects** Spark output if:
- Dimension type mismatch (string where number expected, unless "UNKNOWN")
- Negative dimensions (length/width/height < 0)
- Zero dimensions for solid objects
- Position outside grid bounds
- Conflicting values in same output (e.g., `length: 5` and `length: 10` for same object)

**Rejection Response:**
```json
{
  "status": "rejected",
  "errors": [
    {
      "object_id": "obj_table_01",
      "field": "dimensions.length",
      "reason": "negative_value",
      "value": -3
    }
  ],
  "action": "request_regeneration"
}
```

## 5. Detail Drift Prevention Strategy

### 5.1 The Problem

**Detail Drift:** When Spark regenerates the same scene multiple times, numeric values (dimensions, positions) shift slightly, creating narrative inconsistencies.

**Example:**
- Turn 1: "The table is 4 feet long"
- Turn 5 (regeneration): "The table is 5 feet long"  
- Player confusion: "Did the table change?"

### 5.2 Cache-First Resolution

**Strategy:** Lens maintains a persistent cache of first-seen values per object.

#### Cache Structure
```python
cache = {
    "scene_tavern_main": {
        "obj_table_01": {
            "dimensions.length": (4.0, "[SPARK_GENERATED]", timestamp),
            "dimensions.width": (3.0, "[ASSUMED_STANDARD]", timestamp),
            "position.x": (5, "[SPARK_GENERATED]", timestamp)
        },
        "obj_barrel_02": {
            "dimensions.height": (3.0, "[ASSUMED_STANDARD]", timestamp)
        }
    }
}
```

#### Cache Precedence Rules

1. **First Write Wins:** First non-UNKNOWN value for a field is cached
2. **Cache Overrides Later Unknowns:** If cached, ignore subsequent "UNKNOWN" emissions
3. **Explicit Values Override Cache:** If Spark emits a new explicit value, log conflict but allow update
4. **Cache Persists Across Sessions:** Stored in Lens database

### 5.3 Conflict Detection

When Spark emits a value that contradicts cache:

```python
def check_conflict(object_id, field, new_value):
    cached = cache.get(object_id, field)
    
    if cached and new_value != cached.value and new_value != "UNKNOWN":
        log_warning(
            f"DETAIL_DRIFT detected: {object_id}.{field}",
            f"Cached: {cached.value} ({cached.tag})",
            f"New: {new_value} ([SPARK_GENERATED])",
            f"Action: Preferring cached value"
        )
        return cached.value  # Use cached, ignore new
    
    return new_value
```

### 5.4 Invalidation Strategy

Cache entries are invalidated when:
- Object explicitly destroyed (`condition: "destroyed"`)
- Scene marked as `[RESET]` by DM
- Player explicitly modifies object (e.g., "I cut the table in half")
- TTL exceeded (optional: 7 days of non-access)

### 5.5 Regeneration Best Practices

When Spark regenerates a scene:

1. **Fetch prior Fact Pack from Lens** (if available)
2. **Reuse all prior explicit values** for stable objects
3. **Emit UNKNOWN for any uncertain restatements**
4. **Only change values if narrative demands** (e.g., "the table is now broken")

**Prompt Engineering Example:**
```
SYSTEM: You are regenerating scene_tavern_main. 
Prior fact pack available. REUSE prior dimensions unless 
narrative explicitly changed them. When uncertain, emit UNKNOWN.

Prior objects: obj_table_01 (4ft × 3ft × 2.5ft), obj_barrel_02 (2ft × 2ft × 3ft)
```

## 6. Implementation Recommendations

### 6.1 Pydantic Schema (Python)

```python
from pydantic import BaseModel, Field, field_validator
from typing import Literal, Optional, Union
from enum import Enum

class SizeCategory(str, Enum):
    FINE = "Fine"
    DIMINUTIVE = "Diminutive"
    TINY = "Tiny"
    SMALL = "Small"
    MEDIUM = "Medium"
    LARGE = "Large"
    HUGE = "Huge"
    GARGANTUAN = "Gargantuan"
    COLOSSAL = "Colossal"

class Dimension(BaseModel):
    value: Union[float, Literal["UNKNOWN"]]
    
    @field_validator('value')
    def validate_dimension(cls, v):
        if v != "UNKNOWN" and v < 0:
            raise ValueError("Dimensions cannot be negative")
        return v

class ObjectDimensions(BaseModel):
    length: Union[float, Literal["UNKNOWN"]] = Field(description="Length in feet")
    width: Union[float, Literal["UNKNOWN"]] = Field(description="Width in feet")
    height: Union[float, Literal["UNKNOWN"]] = Field(description="Height in feet")

class Position(BaseModel):
    x: int = Field(ge=0, description="X coordinate (0-indexed grid square)")
    y: int = Field(ge=0, description="Y coordinate (0-indexed grid square)")

class Affordances(BaseModel):
    flammable: bool = False
    climbable: bool = False
    provides_cover: bool = False
```

### 6.3 Lens Validation Pipeline

```python
def validate_and_resolve(fact_pack: SceneFactPack) -> ResolvedSceneFactPack:
    """
    1. Validate schema (Pydantic)
    2. Resolve UNKNOWN values from cache or archetypes
    3. Tag all values with provenance
    4. Detect conflicts
    5. Return resolved + tagged fact pack
    """
    resolved_objects = []
    
    for obj in fact_pack.objects:
        resolved_dims = {}
        for field in ['length', 'width', 'height']:
            value = getattr(obj.dimensions, field)
            resolved_value, tag = resolve_dimension(obj.object_id, f"dimensions.{field}", value, obj.class_)
            resolved_dims[field] = (resolved_value, tag)
        
        resolved_objects.append({
            "object": obj,
            "resolved_dimensions": resolved_dims
        })
    
    return ResolvedSceneFactPack(
        original=fact_pack,
        resolved_objects=resolved_objects,
        validation_timestamp=now(),
        conflicts_detected=conflicts
    )
```

### 6.4 Box Integration Notes

**PropertyMask Compatibility:**
- Spark emits affordances as boolean flags
- Lens converts to PropertyMask uint32 bitmask
- Box reads bitmask for collision/LOS checks

**Example Mapping:**
```python
PROPERTY_MASK_MAPPING = {
    "provides_cover": 0b0001,
    "blocks_los": 0b0010,
    "difficult_terrain": 0b0100,
    "flammable": 0b1000,
    # ... up to 32 properties
}

def affordances_to_mask(affordances: Affordances) -> int:
    mask = 0
    for prop, bit in PROPERTY_MASK_MAPPING.items():
        if getattr(affordances, prop, False):
            mask |= bit
    return mask
```

## Appendix B: Common Pitfalls

### Pitfall 1: Approximate Dimensions
**Bad:** `{"length": "about 5 feet"}`  
**Good:** `{"length": 5}` or `{"length": "UNKNOWN"}`

### Pitfall 2: Metric Units
**Bad:** `{"height": 3}` where 3 is meters  
**Good:** `{"height": 10}` (always feet)

### Pitfall 3: Center vs Corner Positions
**Bad:** Large creature at (5,5) overlaps (5,5), (6,5), (5,6), (6,6)  
**Good:** Specify which square is the "anchor" (top-left by convention)

### Pitfall 4: Regeneration Without Cache
**Bad:** Spark re-emits different dimensions each time  
**Good:** Lens fetches cache, Spark reuses or emits UNKNOWN

### Pitfall 5: Missing Affordances
**Bad:** Stone pillar with no `blocks_los` flag  
**Good:** Pillar marked `blocks_los: true`, `provides_cover: true`, `cover_type: "total"`
