### TD-002: Replay Runner Handles Only 4 of 41+ Event Types
**Status:** DEFERRED — by design
**File:** `aidm/core/replay_runner.py` → `reduce_event()`

**Handled:** `no-op`, `set_entity_field`, `add_entity`, `remove_entity`
**Not handled:** 37+ event types (attack_roll, damage_roll, hp_changed, save_rolled, etc.)

**Why this is intentional:** Combat replay uses **re-execution** (same RNG seed produces same output), NOT event reduction. The replay runner only needs to handle entity management events. Combat events are regenerated by re-running the combat with the same seed.

**Why NOT to fix now:**
- Adding all 37+ event types to reduce_event() would be wasted work
- The re-execution strategy is documented in `AGENT_DEVELOPMENT_GUIDELINES.md` Section 6
- If event replay becomes needed later, it should be a dedicated SKR with design doc

**Correct behavior for agents:** If you add a new entity management event type (add/remove/modify entity fields), add it to reduce_event(). If you add a combat/resolution event type, do NOT add it — it will be replayed via re-execution.

## Category 5: M1 Runtime Implementation Debt

### TD-016: BL-017/018 Violations in runner.py CLI Entry Point
**Status:** ACCEPTED — intentional at CLI boundary
**File:** `aidm/runtime/runner.py`
**Lines:** 207, 297-298

**Violations:**
1. Line 207: `datetime.now(timezone.utc)` in minimal_resolver stub
2. Lines 297-298: `uuid.uuid4()` and `datetime.now(timezone.utc)` in run_session()

**Decision: ACCEPT AS INTENTIONAL**

**Rationale:**
Per OPUS_WO-M1-RUNTIME-02_GUARDRAILS.md Section 1.13:
> "DON'T call `uuid.uuid4()` or `datetime.utcnow()` / `datetime.now()` anywhere in `aidm/runtime/bootstrap.py`. All IDs and timestamps must be injected by the caller (BL-017, BL-018)."

The guardrails specify that bootstrap.py must not generate IDs/timestamps. However, runner.py is the **CLI entry point** — it IS the caller that performs injection. This is the correct boundary for ID/timestamp generation in a CLI context.

**Evidence from design documents:**
1. WO-M1-02 (OPUS_WO-M1-02_inject_only_ids_timestamps.md) states: "All ID and timestamp fields are now required parameters (inject-only). **Callers must explicitly provide values at construction time.**"
2. The inject-only policy applies to schemas and runtime core — not to the ultimate boundary where values must originate.
3. For test harnesses, IDs/timestamps are injected from test code. For CLI usage, they must be generated somewhere — and runner.py is the correct location.

**Why this is acceptable:**
- runner.py is the **entry point** for CLI execution, not library code
- The runtime layer itself (bootstrap.py, session.py) correctly requires injected values
- This design allows test harnesses to inject deterministic values while allowing CLI to generate real values
- The minimal_resolver stub (line 207) is explicitly documented as "M1.5 vertical slice stub" — real resolvers would receive injected resolved_at

**What would NOT be acceptable:**
- Generating IDs/timestamps in bootstrap.py (violates BL-017/018)
- Generating IDs/timestamps in session.py core logic (violates BL-017/018)
- Generating IDs/timestamps in schema default_factory (already prohibited by WO-M1-02)
- Generating IDs/timestamps in resolvers without injection interface (design smell)

**Future work (if needed):**
- If CLI grows complex enough to need testing with deterministic IDs/timestamps, extract a `CLIOrchestrator` class that accepts an optional ID/timestamp generator function
- For now, the simple inline generation is appropriate for the M1.5 minimal vertical slice
