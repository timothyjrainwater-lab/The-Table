# 2) System Map (As understood)

## Architecture bullets

* **Box**: deterministic rules engine; authoritative state transitions; event-sourced log; replay via re-execution with seeded RNG. *(PSD “Core Engine,” “Combat Resolution Stack,” TD-002.)*
* **Lens**: containment membrane; builds `NarrativeBrief` from Box events; token-budget context assembly; now also owns voice characterization mediation (planned). *(PSD WO-032; WO-052.)*
* **Spark**: swappable LLM providers via external registry; supports optional JSON mode; kill-switch / grammar shield gating. *(PSD mentions Spark tests; `T2_SPARK_PROVIDER_CONTRACT.md`.)*
* **Immersion**: STT/TTS/Image/AudioMixer adapters; must be non-authoritative; pure deterministic compute functions; stub-by-default and lazy-loading. *(PSD “M3 Immersion Layer v1”; `T3_IMMERSION_BOUNDARY.md`.)*
* **Runtime**: session bootstrap + orchestrator conducts turn cycle: STT → IntentBridge → Box execute_turn → NarrativeBrief → DM persona prompt → narration → TTS; fallbacks for STT/TTS/Spark. *(PSD WO-039/045/046 and P3 fallback.)*

## Authority boundaries

* **Box is authoritative** over mechanics and world state mutation. *(PSD, doctrine.)*
* **Lens/Spark/Immersion are non-authoritative**: may generate narration/media, but must not influence mechanics. *(Coherence doctrine; immersion boundary contract.)*
* **Runtime is the conductor** but must preserve the boundary: it routes inputs, validates, and logs. *(PSD runtime/orchestrator descriptions.)*

## Critical data flows

* **Input → Intent**: STT/text command → IntentBridge (name resolution / ambiguity) → canonical intents. *(PSD WO-038/039.)*
* **Intent → Mechanics**: `play_loop.execute_turn()` emits Box events + state mutations. *(PSD Box integration.)*
* **Mechanics → Narrative**: Box events → typed event contracts → `NarrativeBrief` → Spark prompt. *(PSD WO-046 + WO-032.)*
* **Narrative → Immersion**: narration text + `VoicePersona` → TTS adapter. *(PSD + WO-051/052.)*
* **Replay**: re-execute deterministic mechanics from event log + RNG seed; some payload equality tests exist, with explicit exclusion of nondeterministic correlation IDs. *(PSD P1 + TD-002.)*

# 3) Gap Register (Prioritized)

### GAP-001

* **Severity:** BLOCKER
* **Area:** docs / ops / UX
* **Symptom:** Multiple “canonical” documents conflict on what is implemented (e.g., “no ASR/TTS implementation” vs orchestrator uses STT/TTS; Vertical Slice says “no combat resolver” vs PSD shows full combat stack).
* **Impact:** Engineers and auditors will make incorrect assumptions; test/feature expectations diverge; onboarding breaks; risk of reintroducing “fixed” problems.
* **Evidence:** `T1_PROJECT_STATE_DIGEST.md` (Phase 3 COMPLETE; STT/TTS usage), `T1_PROJECT_COHERENCE_DOCTRINE.md` (Out-of-scope: “Production voice integration”; test runtime gate), `T2_VERTICAL_SLICE_V1.md` (scope excludes combat resolver).
* **Likely root cause:** Documentation lifecycle not governed; “single source of truth” not enforced; older docs not explicitly deprecated.
* **Fix direction:** Declare explicit doc hierarchy + deprecation banners; update doctrine to match PSD or mark it historical; add a “DOC_DRIFT.md” ledger and a “current architecture” one-pager pinned to PSD.
* **Acceptance criteria:** A new “Current System Snapshot” doc aligns with PSD; old docs have prominent “HISTORICAL / SUPERSEDED” headers; onboarding checklist points to the right sources.
* **Stop condition:** If maintainers cannot agree on the doc hierarchy within one review cycle, freeze changes and escalate to PO/PM for a binding doctrine update.

### GAP-002

* **Severity:** HIGH
* **Area:** tests / ops
* **Symptom:** Test runtime gate in doctrine (<5s) is violated (PSD reports ~51s) without a revised enforcement policy.
* **Impact:** CI scalability risk; contributors will optimize for speed differently; the gate becomes meaningless and future regressions are undetectable.
* **Evidence:** `T1_PROJECT_COHERENCE_DOCTRINE.md` §3 vs `T1_PROJECT_STATE_DIGEST.md` Test Count note.
* **Likely root cause:** The gate was set early and never renegotiated; suite growth outpaced governance.
* **Fix direction:** Re-baseline test policy: define per-test average target, tiered suite (fast vs slow), and hardware-gated separation.
* **Acceptance criteria:** Updated governance doc that matches actual suite behavior; CI config (if any) reflects tiers; “fast gate” is enforced.
* **Stop condition:** If the project refuses to adjust governance while continuing to add tests, audits should treat “test performance” as ungoverned.

### GAP-003

* **Severity:** HIGH
* **Area:** determinism / integration
* **Symptom:** Non-deterministic IDs exist in event payloads (uuid4) and are excluded from equality comparisons; unclear if any downstream component relies on these IDs for replay correctness or persistence.
* **Impact:** Subtle replay drift, flaky integration tests, and broken “show your work” receipts if correlation IDs are used as stable references.
* **Evidence:** `T1_PROJECT_STATE_DIGEST.md` P1 Replay Stability finding (“spell_cast cast_id uses uuid.uuid4() … excluded”).
* **Likely root cause:** Convenience correlation IDs added at integration time; determinism contract applied only to state hash, not full payload semantics.
* **Fix direction:** Document a strict “nondeterministic fields allowlist” and enforce it centrally; ensure IDs used in receipts and storage are injected deterministically or derived from event content.
* **Acceptance criteria:** A single authoritative list of nondeterministic fields; replay/payload tests reference the allowlist; no nondeterministic field is ever used as a persistence key.
* **Stop condition:** If additional nondeterministic fields are added ad hoc, halt integration until the allowlist is enforced.

### GAP-004

* **Severity:** HIGH
* **Area:** UX / ops
* **Symptom:** “Real user” runbook is missing: how to install deps, choose models, run a session, handle missing backends, and recover after failure.
* **Impact:** The system remains “developer-only”; playtests will fail due to setup friction, not product issues.
* **Evidence:** PSD describes many components but does not provide operator steps; `T3_KNOWN_TECH_DEBT.md` TD-016 frames runner.py as injection boundary without a user-facing procedure.
* **Likely root cause:** Engineering-focused milestones; packaging and runbooks deferred.
* **Fix direction:** Produce an “Operator Quickstart” with deterministic example session, expected outputs, and failure recovery table.
* **Acceptance criteria:** A new runbook that a clean machine can follow; scripted smoke run that produces a log + replay verification + narration fallback demonstration.
* **Stop condition:** If Phase 4 playtests begin without this, treat failures as non-diagnostic.

### GAP-005

* **Severity:** HIGH
* **Area:** UX / immersion
* **Symptom:** Voice is positioned as central, but there is no operational voice roster (no legal reference clip sourcing, no resolver integration yet).
* **Impact:** Player-perceived sameness; undermines “voice-first” premise; reduces perceived intelligence of the system even if mechanics are correct.
* **Evidence:** `T3_WO-051_VOICE_DESIGN_RESEARCH.md` (no reference clips; “every character sounds the same”), `T3_WO-052_LENS_VOICE_RESOLVER.md` (depends on WO-051).
* **Likely root cause:** Research and content sourcing deferred; integration gated behind artifacts not yet produced.
* **Fix direction:** Treat reference-clip roster as a product dependency (content pipeline), not optional research; define minimum viable roster and licensing constraints.
* **Acceptance criteria:** A starter roster (10–15) with validated licenses + a deterministic mapping policy; demo scenario uses distinct voices.
* **Stop condition:** If voice-first playtests proceed without a roster, stop and re-scope the experience claims.

### GAP-006

* **Severity:** MEDIUM
* **Area:** requirements / UX
* **Symptom:** Spark provider contract includes capabilities (seeded generation, JSON mode), but the player experience around variability is not specified (cache vs regenerate, replay narration expectations).
* **Impact:** Confusion: “replay verified” but narration differs; player distrust; bug reports that are “expected behavior.”
* **Evidence:** `T2_SPARK_PROVIDER_CONTRACT.md` (seed field; JSON mode), PSD replay strategy emphasis.
* **Likely root cause:** Mechanical determinism treated as the only determinism that matters; narrative determinism not product-defined.
* **Fix direction:** Define “Narrative Determinism Policy”: what must match on replay (mechanics) and what may vary (narration), and whether narration is cached per session.
* **Acceptance criteria:** Policy doc + a test or demo that explicitly shows “mechanics identical / narration may vary” with clear messaging.
* **Stop condition:** If support load increases due to this ambiguity, halt new Spark features until policy exists.

### GAP-007

* **Severity:** MEDIUM
* **Area:** integration / boundary
* **Symptom:** Replay runner intentionally reduces only entity-management events; combat replay is re-execution. This is fine, but any future feature that requires “event reduction replay” will silently break unless gated.
* **Impact:** Future additions (e.g., external tools reading event logs) will assume event logs are reducible; incompatible expectations create costly rewrites.
* **Evidence:** `T3_KNOWN_TECH_DEBT.md` TD-002.
* **Likely root cause:** Two replay mental models coexist: event reduction vs deterministic re-execution.
* **Fix direction:** Document “Replay Semantics” as a contract: which events are authoritative inputs vs derived outputs; how third parties should interpret logs.
* **Acceptance criteria:** A short spec that clearly labels reducible events and derived events; tooling/tests reflect this.
* **Stop condition:** If anyone proposes adding dozens of reducers “because replay,” escalate—this is explicitly deferred.
