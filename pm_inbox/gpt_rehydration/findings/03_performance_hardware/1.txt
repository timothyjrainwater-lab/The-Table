### RG-003 — Voice pipeline foundation: reference audio corpus + sourcing policy (WO-051 prerequisite)

* **Question:** What is the *reference audio strategy* (number of clips, duration, licensing, storage layout, normalization) to enable voice cloning and character differentiation?
* **Source:** WO-052 explicitly depends on WO-051 research and notes current “every character uses the same default persona.”

* **Research Type:** **Empirical + Community + Decision**

* **Estimated Scope:** **Large (weeks)**

### RG-004 — VRAM residency + unload/reload policy across Spark / SDXL / Music

* **Question:** What is the enforced runtime policy for model residency (Spark vs SDXL vs ACE-Step) to prevent OOM and maintain predictable latency?
* **Source:** Spark contract includes OOM fallback behavior and model switching guidance. 
  Tech stack/VRAM assumptions appear in PM rehydration. 
* **Blocks:** Any “full loop” immersion usage where multiple GPU-heavy backends compete; also impacts stress-test realism.
* **Research Type:** **Empirical + Design decision**
* **Recommended Approach:** Benchmark GPU memory footprints and cold-start latencies per backend on target hardware; decide a single arbitration policy (priority order + prefetch points + eviction triggers) aligned with “prep-first” sequencing.
* **Estimated Scope:** **Medium (days)**

### RG-005 — Canonical ID schema and rename/delete semantics at Lens boundaries

* **Question:** What are the invariant rules for entity IDs across SessionBundle → SceneCards → Box events → Lens caches, including **rename**, **merge**, and **delete** semantics?
* **Source:** Vertical Slice V1 shows SessionBundle/SceneCard IDs but does not define lifecycle/renames. 
  Spark/Box integration relies on strict routing and validation. 
* **Blocks:** Long-running sessions with persistence, cache invalidation, and anything using “voice roster per character_id” (WO-052 introduces a VoiceRoster keyed by character_id). 
* **Research Type:** **Design decision**
* **Recommended Approach:** Extract all current ID usages from schemas + runtime components, then write a single canonical “ID & Lifecycle Contract” (rename/delete rules + migration behavior + hash stability implications).
* **Estimated Scope:** **Medium (days)**

## Section 2: Design Decisions Needed (with trade-offs)

### DD-01 — “Structured output” strategy: native JSON mode vs guided generation + parsing

* **Decision:** Standardize on either (A) native JSON mode where available, or (B) guided prompting + Lens parsing as the universal baseline.
* **Trade-off:**

  * **A (native JSON):** cleaner, fewer parse failures, but uneven provider support and higher coupling to backend quirks. 
  * **B (guided + parse):** portable and already specified as fallback, but needs robust repair/retry policy and increases Lens complexity. 
* **Why it matters:** This decision determines the final form of RG-001’s “Spark Emission Contract.”

### DD-02 — Context overflow handling: truncate vs summarize vs user-approved model switch

* **Decision:** Pick the default overflow strategy and when escalation is allowed.
* **Trade-off:**

  * Truncate oldest: deterministic and cheap, but can lose crucial narrative continuity. 
  * Summarize: preserves info but costs extra calls and risks summary drift. 
  * Model switch: preserves full context but requires UX + user approval flow. 

### DD-03 — Voice resolver determinism vs variety

* **Decision:** Should voice characterization be stable across sessions for the same NPC archetype, or re-generated per introduction?
* **Trade-off:**

  * Stable mapping improves recognition; reduces creative variance.
  * Re-generation increases variety but risks “character voice drift.”
* **Anchors:** WO-052 explicitly requires “deterministic enough to produce consistent results across LLM calls” and a caching roster.
