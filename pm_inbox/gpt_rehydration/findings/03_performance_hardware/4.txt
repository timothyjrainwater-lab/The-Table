## WP-01 (F1): Execute and Capture GPU Performance Gate

**Objective**
Produce authoritative, reproducible GPU performance evidence for Spark runtime.

**Scope Boundaries**
IN: Spark stress suite, GPU-gated tests, real hardware measurement, artifact capture.
OUT: Changing model architecture or tuning prompts.

**Artifacts Touched**

* `T1_PROJECT_STATE_DIGEST.md` (update with measured results)
* `tests/test_spark_integration_stress.py` (ensure GPU tests are runnable)
* Perf artifact directory (e.g., `artifacts/perf/gpu_run_YYYYMMDD.json`)
* Release checklist doc

**Step Plan**

1. Identify canonical GPU rig (target 3080 Ti class).
2. Enable hardware-gated tests in stress suite.
3. Run 30–50 turn narration cycles.
4. Record:

   * Peak VRAM per turn
   * Cold-start time
   * Reload time
   * Tokens/sec
   * p50/p95 latency
5. Repeat under fallback model scenario.
6. Repeat under forced overflow condition.
7. Save raw logs + summarized JSON artifact.
8. Attach artifact to release gate.

**Acceptance Criteria**

* No OOM under defined test scenario.
* p95 latency recorded for each path.
* Artifact reproducible within ±10%.

**New Tests Required**

* `test_gpu_perf_baseline_recording`

  * Proves measurement harness works.
* `test_gpu_reload_latency_bounds`

  * Proves reload < defined threshold.

**Rollback Plan**
Revert any temporary instrumentation if it contaminates determinism.

**Stop Conditions**

* VRAM exceeds GPU capacity.
* Reload latency > 5× acceptable target.
  Escalate to F2 immediately.

# 3) Single Highest-Risk Unknown

### Highest-Risk Unknown

**Whether GPU unload/reload actually reclaims VRAM reliably and cheaply under real driver/runtime conditions.**

If unload does not reclaim memory deterministically, the entire VRAM arbitration plan collapses and OOM fallback becomes unstable.

### Fastest Resolution Path

1. On target GPU rig:

   * Load primary Spark model.
   * Record VRAM.
   * Unload.
   * Record VRAM.
   * Repeat 20 cycles.
2. Force OOM and fallback.
3. Track VRAM after each cycle.

**What to inspect**

* Peak VRAM
* Post-unload VRAM baseline
* Fragmentation pattern
* Latency increase trend

This resolves the core feasibility question in under one day and determines whether the current architecture can survive real-world memory pressure.
