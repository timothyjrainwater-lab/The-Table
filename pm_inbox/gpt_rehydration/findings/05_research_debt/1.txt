# AIDM Research Gap Register

## Section 1: Critical Path Blockers

### RG-001 — Voice Differentiation: reference audio sourcing and standardization

* **Question:** What is the canonical strategy to obtain, license, normalize, and package *reference audio clips* so Chatterbox-style voice cloning yields distinct, stable character voices?
* **Source:** Voice pipeline explicitly needs references; resolver depends on WO-051 research output for archetype recipes and mapping tables.
* **Blocks:** Practical voice pipeline usefulness (distinct NPCs) and any downstream “voice-first DM” experience quality; also blocks completing WO-052 with real reference selection behavior beyond placeholders.
* **Research Type:** **Community + Literature + Prototype**
* **Recommended Approach:** Survey best practices for voice-cloning reference packs (length, noise floor, phoneme coverage, loudness normalization, file formats, licensing). Prototype a minimal “reference pack” with 5–10 archetypes and run perceptual A/B checks (distinctness, consistency) plus failure-case checks (accent drift, gender/age confusion).
* **Estimated Scope:** **Large (weeks)**

### RG-002 — Deterministic VoicePersona resolution from “creative” descriptors

* **Question:** How do we make LLM-produced voice descriptors *deterministic enough* to map reliably into numeric TTS controls (pitch/speed/exaggeration) while preserving variety?
* **Source:** WO-052 requires a structured prompt with fixed fields and stresses determinism/consistency across LLM calls; it must map to VoicePersona fields and reference selection.
* **Blocks:** Finishing Lens “voice resolver” contract and ensuring stable voices per character within a session (VoiceRoster cache).
* **Research Type:** **Design Decision + Empirical**
* **Recommended Approach:** Define a constrained descriptor vocabulary (already sketched: pitch/timbre/pace/etc.), then empirically test: (a) intra-model repeatability under fixed seed/system prompt, (b) mapping collision rate (different characters → same persona), (c) “drift” rate over multiple calls. Decide acceptable collision/drift thresholds.
* **Estimated Scope:** **Medium (days)**

### RG-003 — Spark/Narration research synthesis is incomplete (RQ-SPARK-001 / RQ-NARR-001)

* **Question:** What is the consolidated, canonical conclusion of the delivered sub-findings, and what rules/policies become binding for implementation?
* **Source:** Opus PM rehydration explicitly states all 3 sub-dispatches for both RQs are delivered, but synthesis is still “ready to compile.”
* **Blocks:** Any “Phase 4 ship gate” readiness arguments that rely on a validated narration policy and structured fact emission policy; also blocks reducing risk of ad-hoc narration behavior under stress. (Even if code exists, the *policy* remains non-canonical until synthesized.)
* **Research Type:** **Synthesis (Design Decision)**
* **Recommended Approach:** Compile a single synthesis doc per RQ: reconcile contradictions across A/B/C findings, extract the final invariant set (must/should/must-not), and produce testable acceptance criteria (golden outputs + adversarial cases).
* **Estimated Scope:** **Small–Medium (hours–days)**

### RG-004 — Capability-gated mechanics: missing prerequisite CPs for AoO provocation cases

* **Question:** What is the canonical design for ranged attack intents and spellcasting intents sufficient to implement AoO provocation for ranged attacks and spellcasting?
* **Source:** Known Tech Debt lists AoO provocation TODOs blocked by CP-18A (spellcasting intents) and “weapon/equipment system (not yet designed).”
* **Blocks:** Correct AoO behavior beyond movement triggers; any feats/skills/spells that rely on provocation rules in these cases (and any UX that claims full 3.5e AoO coverage).
* **Research Type:** **Design Decision**
* **Recommended Approach:** Extract minimal intent schemas needed (fields, validation rules) and define the equipment/weapon parameter source-of-truth required for provocation decisions. Confirm against SRD/PHB rules citations used by the project’s source registry approach.
* **Estimated Scope:** **Medium (days)**

### RG-005 — Permanent stats integration with temporary modifiers

* **Question:** What is the precise integration contract between permanent stat computation and temporary condition modifiers (e.g., Bull’s Strength), and how is ordering/stacking validated?
* **Source:** Known Tech Debt flags a stub in permanent_stats (“temp_total = 0”) blocked by CP-16 integration; warns not to wire without a dedicated CP/tests.
* **Blocks:** Accurate ability score–dependent mechanics once temporary buffs/debuffs are introduced at scale; reduces confidence in “permanent stats Phase 3 complete” for real play loops involving buffs. (Current tests pass because scenarios exclude it.)
* **Research Type:** **Design Decision + Empirical**
* **Recommended Approach:** Specify modifier precedence/stacking and the exact data flow from condition system into stat computation (pure inputs). Validate with a focused matrix of canonical D&D cases and property tests (monotonicity, idempotence).
* **Estimated Scope:** **Medium (days)**

## Section 2: Design Decisions Needed (with trade-offs)

### DD-001 — Voice reference selection: string matching vs embeddings vs fixed roster

* **Decision:** WO-052 currently constrains reference selection to **string matching, not embedding similarity** (“keep it simple”).
* **Tension:**

  * *String matching* is deterministic, cheap, and testable; but brittle to phrasing variety (LLM synonyms, multi-attribute descriptions).
  * *Embeddings* improve semantic matching but introduce model dependency, compute cost, and new nondeterminism surfaces.
* **Recommendation (research framing):** Keep the constraint unless voice collisions become unacceptable; if collisions are high, consider a **two-stage** approach: strict structured fields → deterministic canonical tags → string match tags (not free text).

### DD-002 — Where “voice characterization” lives: Spark prompt policy vs Lens mapping policy

* **Decision:** Lens must not call Spark inside resolver; it should generate prompts/parse responses, remaining a pure mapping layer with session-scoped caching.
* **Tension:**

  * Pushing logic into Spark increases creativity but increases drift and policy surface.
  * Keeping mapping in Lens improves determinism but can feel “same-y” unless archetype library is rich (blocked by RG-001).

### DD-003 — “Unknown backend” behavior for immersion adapters

* **Decision:** Immersion factories must raise ValueError for unknown backends and ImportError for missing libs; stubs must always exist; caller continues unaffected by adapter failure.
* **Trade-off:** Strong fail-closed semantics for configuration vs user-friendly “auto fallback.” The contract already chooses explicit failure for misconfig while still guaranteeing “engine continues unaffected.”

## Section 3: Empirical Unknowns (benchmarks / validation)

### EU-001 — GPU-skipped Spark stress tests: real-hardware performance envelope

* **Unknown:** WO-033 Spark Integration Stress Test has **GPU-gated skips** (4 tests skipped), implying the “real hardware” envelope still needs measurement for those scenarios.
* **Research Type:** **Empirical**
* **Approach:** Run the skipped GPU tests on the target 3080 Ti environment; record VRAM usage, latency distributions, and failure rates under repeated runs (thermal + fragmentation effects).

### EU-002 — VoicePersona distinctness: perceptual thresholds

* **Unknown:** Even with mapping, what measurable threshold defines “characters do not all sound the same” (MOS-like ratings, ABX success rate)?
* **Research Type:** **Empirical**
* **Approach:** Define a small human evaluation protocol (5–10 listeners or the owner + a rubric): distinctness, consistency over time, and appropriateness vs character context.

## Section 5: Implicit Assumptions (hand-waves / undefined protocols)

### IA-001 — “Spark produces consistent structured voice fields” is assumed

* **Assumption:** The structured prompt will reliably return the exact field set with acceptable stability. The WO asserts the requirement but doesn’t provide demonstrated compliance criteria or a repair strategy for partial/malformed responses.
* **Risk:** Downstream mapping becomes error-prone; fallback behavior becomes policy-heavy.

### IA-002 — “Engine continues unaffected” under immersion failures depends on caller behavior

* **Assumption:** Because immersion adapters must not block engine execution and must not write AssetStore directly, the orchestration layer will always honor that separation (not shown here; only the boundary contract is explicit).

### IA-003 — Deferred mechanics are safe because tests don’t cover those scenarios

* **Assumption:** Stubs in permanent stats and AoO provocation are safe under current legal capabilities; this holds as long as capability gates stay enforced and no scenario leaks into those mechanics. The docs warn not to “fix” them without CPs, but don’t enumerate detection/guard rails beyond process discipline.

## Section 6: Research Synthesis Opportunities (RQ-SPARK-001 / RQ-NARR-001)

Both RQs have delivered sub-findings but lack canonical synthesis.

### What a proper synthesis doc should contain (per RQ)

1. **Problem statement + threat model** (what failure looks like under stress; what “fail-closed” means for narrative outputs).
2. **Final binding policies** (must/should/must-not), including:

   * output format constraints,
   * repair loops (if any),
   * kill-switch triggers tied to observable violations.
3. **Acceptance tests**: golden outputs + adversarial cases + determinism proofs (where applicable).
4. **Integration contract**: which layer owns which responsibilities (Spark vs Lens vs Guard services).
5. **Decision log**: resolved trade-offs and rejected alternatives.
