## Section 1: Critical Path Blockers

### RG-001 — “Document freshness / authority conflict” (stale plan vs current state)

* **Question:** Which documents are authoritative *today* for “what exists” vs “what was planned,” and how do we prevent stale specs from reintroducing already-solved gaps?
* **Source:** Execution plan draft claims core systems have “zero code / stubs only” and calls RQ-SPARK-001/RQ-NARR-001 “NOT DELIVERED.” vs PSD shows Phase 2/3 WOs integrated and A10 passed.
* **Blocks:** Reliable research audit, correct dispatching, avoiding duplicate investigations.
* **Research Type:** **Decision** (governance) + **Community** (team process)
* **Recommended Approach:** Establish a “source of truth” precedence list (PSD > code/tests > README > doctrine > historical plans). Add a lightweight “doc validity header” rule (status + last-validated-against-PSD date).
* **Estimated Scope:** **Small** (hours)

### RG-002 — Voice uniqueness is blocked by missing reference roster + mapping strategy

* **Question:** What legally usable reference-clip roster (10–15 voices) and archetype parameter recipes should be used so NPCs don’t collapse to the default voice?
* **Source:** WO-051 explicitly states adapter works but “no reference clips,” and defines required deliverable (voice design guide, sourcing, recipes).
* **Blocks:** WO-052 voice resolver (depends on WO-051 output) and any voice-forward play experience.
* **Research Type:** **Literature + Community + Prototype** (light prototyping via small clip set evaluation, not product code)
* **Recommended Approach:** Execute WO-051 as written: survey datasets (VCTK/LibriSpeech/Common Voice/etc.), extract 10–15s clean clips, and build a recipe table mapping archetypes to VoicePersona ranges and clip selection heuristics.
* **Estimated Scope:** **Medium** (days)

### RG-003 — Voice-to-character determinism + caching semantics (Lens voice resolver)

* **Question:** How do we ensure voice assignment is consistent (deterministic enough) across sessions/turns, and what are the cache invalidation rules when character descriptions evolve?
* **Source:** WO-052 requires prompt determinism and a session-scoped cache, but does not define *stability targets*, drift handling, or “description changed” policy.
* **Blocks:** Reliable character identity continuity in narration/voice; prevents “voice jitter.”
* **Research Type:** **Design decision + Empirical**
* **Recommended Approach:** Define explicit acceptance metrics (e.g., same character_id ⇒ same persona_id for entire session unless explicit override; bounded variance thresholds if Spark output is used). Then run a small prompt-repeatability study on Qwen3 variants.
* **Estimated Scope:** **Medium** (days)

### RG-004 — Spark↔VRAM residency policy (LLM + SDXL can’t co-reside)

* **Question:** What is the *runtime residency policy* for GPU models when Spark + SDXL cannot load simultaneously (eviction order, prefetching, turn budget)?
* **Source:** Hardware constraint described in project context; immersion is atmospheric and excluded from replay, but the *resource arbitration policy* is not captured in the schema docs shown.
* **Blocks:** Stable “prep-phase” pipeline timing and user experience under real hardware constraints; prevents thrash.
* **Research Type:** **Empirical + Design decision**
* **Recommended Approach:** Benchmark load/unload latency and steady-state inference times across Qwen3 sizes and SDXL Lightning NF4; derive a simple state machine policy (single-GPU residency).
* **Estimated Scope:** **Medium–Large** (days to weeks depending on breadth)

### RG-005 — “Immersion is non-authoritative” enforcement gaps for voice/STT intents

* **Question:** What is the *fail-closed* policy when STT confidence is low, ambiguous, or conflicts with deterministic constraints—without letting immersion become mechanical authority?
* **Source:** Immersion schemas explicitly state atmospheric-only and “never mechanical authority.” Development guidelines warn about intent-schema pitfalls and voice-vs-combat intent confusion.
* **Blocks:** Voice-first usability and safety (incorrect actions), especially with intent bridge.
* **Research Type:** **Design decision + Empirical**
* **Recommended Approach:** Define an explicit “confidence gates → clarification loop” policy (declare→point→confirm) tied to intent schemas; measure with faster-whisper confidence distributions in noisy conditions.
* **Estimated Scope:** **Medium** (days)

## Section 2: Design Decisions Needed (with trade-offs)

### DD-01 — Voice persona resolution: rule-based vs Spark-described

* **Tension:** WO-052 proposes Spark-produced descriptors then Lens maps to numeric persona fields. That increases flavor but risks nondeterministic drift.
* **Options:**

  1. **Deterministic lookup-first** (character tags → archetype → fixed persona) + Spark only for *new* NPCs.

     * Pros: stable, debuggable. Cons: less variety.
  2. **Spark-first** (LLM describes every new speaker) + strict cache.

     * Pros: expressive. Cons: drift risk; repeatability demands.
* **Decision needed:** Target level of variability allowed per session and acceptable failure modes (fallback persona vs “ask again”).

### DD-02 — Voice roster licensing strategy

* **Tension:** WO-051 lists multiple sources (public domain audiobooks, datasets, CC reels, self-recording, AI-generated seeds).
* **Decision needed:** “Shipping roster” must have a compliance posture: *redistributable by default* vs *user-provided only*.

### DD-03 — Session persistence of voice roster

* **Tension:** WO-052 explicitly says cache is session-scoped and not persisted. That conflicts with long-running campaigns where NPC continuity matters.
* **Decision needed:** Persist persona assignment (persona_id + reference_audio pointer + params) as metadata *without* affecting deterministic combat replay.
