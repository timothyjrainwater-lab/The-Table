## Box ‚Üî Lens seam (mechanical truth ‚Üí narrative-safe context)

### Seam definition (what exists)

* Lens receives Box outputs via a **one-way valve** and produces **NarrativeBrief** (frozen, redacted) for Spark.
* PSD states the intended containment rules explicitly: **no entity IDs, no raw HP/AC/bonuses, no grid coordinates**; HP becomes a **severity bucket** (minor‚Üílethal). (PSD: WO-032)

### Handoff gaps (where the protocol is still ‚Äúhand-waved‚Äù)

**BL-01 ‚Äî ‚ÄúWhat exactly is the Box‚ÜíLens payload?‚Äù**

* The docs say ‚ÄúBox STPs ‚Üí NarrativeBrief,‚Äù but do not define a canonical **input structure** for STPs at the seam:

  * Is it the **event log**? A **turn summary**? A **receipt bundle**? A **state snapshot**?
  * If multiple exist, **which is authoritative** for narration context?
* Consequence: different callers will assemble different ‚Äútruth packets,‚Äù breaking consistency and making Grammar Shield rules hard to write.

**BL-02 ‚Äî ‚ÄúFreeze semantics and timing‚Äù**

* Boundary law BL-020 says non-engine modules get **FrozenWorldStateView** only (read-only). (T1_AGENT_DEVELOPMENT_GUIDELINES.md: BL-020)
* What‚Äôs not specified: *when* the snapshot is taken relative to:

  * event commit
  * reducer application
  * multi-event turns (AoO interrupts etc.)
* Consequence: narration can describe an **intermediate** state unless the freeze point is explicitly standardized (‚Äúpost-reducer, post-commit, after all triggered events settle‚Äù).

**BL-03 ‚Äî ‚ÄúRedaction rules beyond the obvious‚Äù**

* WO-032 defines some redactions (IDs, raw HP/AC/coords), but there‚Äôs no full policy on:

  * **spell names** vs ‚Äúa burst of flame‚Äù (are mechanics-bearing names allowed?)
  * **distance** (‚Äú30 feet‚Äù) vs narrative (‚Äúacross the room‚Äù)
  * **status conditions** (Prone/Stunned etc.)‚Äîallowed verbatim, paraphrased, or severity-mapped?
* Consequence: Spark can leak mechanical precision in ways that feel like ‚Äúcombat loop authority,‚Äù even if it isn‚Äôt mutating state.

**BL-04 ‚Äî ‚ÄúProvenance/citation propagation to Lens outputs‚Äù**

* Coherence doctrine says rulings must be citation-backed; events carry citations; rule lookup yields snippets. (T1_PROJECT_COHERENCE_DOCTRINE.md: Provenance/Citations)
* But the seam doesn‚Äôt specify whether NarrativeBrief carries:

  * citations attached to facts it mentions
  * a list of ‚Äúeligible citations for this turn‚Äù
  * or nothing (and Spark is expected to speak without citation context)
* Consequence: ‚ÄúShow your work‚Äù capability becomes inconsistent at the narration layer.

## Lens ‚Üî Spark seam (structured context ‚Üí untrusted generation)

### Seam definition (what exists)

* Spark Provider Contract defines **only** a generic `SparkRequest(prompt, temperature, max_tokens, ‚Ä¶)` and a `SparkResponse(text, finish_reason, tokens_used, ‚Ä¶)`.
* It explicitly assigns ‚ÄúLens/Box Integration‚Äù to AIDM (provider-agnostic responsibility), but does not define **Lens‚Äôs canonical prompt pack format**. (T2_SPARK_PROVIDER_CONTRACT.md: Responsibilities)

### Handoff gaps (protocol undefined)

**LS-01 ‚Äî ‚ÄúCanonical prompt pack schema is missing‚Äù**

* Contract says adapter does ‚Äúprompt formatting,‚Äù but **what Lens passes as ‚Äúprompt‚Äù** is not standardized:

  * Is it a single text prompt?
  * A structured JSON object embedded in text?
  * A multi-section template (‚ÄúFacts / Constraints / Style / Output schema‚Äù)?
* Consequence: Spark behavior varies wildly across call sites; Grammar Shield rules drift; regression testing becomes fragile.

**LS-02 ‚Äî ‚ÄúToken budget allocation policy (hard constraint, no policy)‚Äù**

* PSD mentions ContextAssembler with token budgeting (word_count * 1.3 estimation). (PSD: WO-032)
* Missing: a **budget partition** across:

  * latest turn events
  * scene context (SceneCard)
  * campaign memory (ledger/evidence/threads)
  * DM persona/tone config
* Consequence: different assemblies will drop different context under truncation; narrations become inconsistent and hard to debug.

**LS-03 ‚Äî ‚ÄúOutput schema: free text vs structured emission‚Äù**

* Spark contract supports `json_mode` and discusses ‚Äúparse JSON from text (LENS validates), retry or template fallback.‚Äù (T2_SPARK_PROVIDER_CONTRACT.md)
* Missing: a canonical **SparkResponse payload schema** when structured output is desired:

  * narration text
  * speaker tags (DM vs NPC)
  * per-sentence uncertainty/provenance tags
  * optional ‚Äúimmersion requests‚Äù (images/audio)
* Consequence: there‚Äôs no stable place to hang Grammar Shield validation beyond coarse JSON parsing.

**LS-04 ‚Äî ‚ÄúFailure taxonomy and retry policy at the seam‚Äù**

* Spark contract says ‚Äúretry with corrective prompt OR use template fallback,‚Äù but doesn‚Äôt define:

  * max retries per turn
  * what constitutes a ‚Äúcorrective prompt‚Äù
  * which failures are **fatal** vs **degrade**
* Consequence: unpredictable latency spikes and inconsistent fail-closed behavior at the narration boundary.

**LS-05 ‚Äî ‚ÄúProvenance labeling is asserted but not specified‚Äù**

* Spark contract says AIDM must do provenance labeling (`[NARRATIVE]` / `[UNCERTAIN]`). (T2_SPARK_PROVIDER_CONTRACT.md)
* Missing: the *mechanics*:

  * Is it inline tags, structured fields, or metadata attached to segments?
  * What triggers `[UNCERTAIN]`? Missing facts? Low confidence? Unsupported claims?
* Consequence: ‚Äútrust UX‚Äù becomes subjective and non-testable.

## Spark ‚Üî Immersion seam (narration intent ‚Üí TTS / image / audio actions)

### Seam definition (what exists)

* Immersion boundary contract is clear: read-only, deterministic, no RNG, no engine blocking, caller orchestrates storage. (T3_IMMERSION_BOUNDARY.md)
* Immersion schema defines:

  * `VoicePersona` (TTS config, optional reference clip)
  * `SceneAudioState` (mood + active tracks computed deterministically)
  * `ImageRequest` / `ImageResult`
  * `Transcript` (STT output, atmospheric)
  * `AttributionLedger` (provenance for assets)
  * **Notably missing**: a generalized ‚ÄúAudioRequest‚Äù / ‚ÄúMusicRequest‚Äù schema‚Äîonly ‚Äúcomputed tracks‚Äù exist. (T4_immersion_schema.py)

### Handoff gaps (undefined orchestration contract)

**SI-01 ‚Äî ‚ÄúWho decides what immersion assets to generate?‚Äù**

* Immersion says ‚Äúcaller orchestrates AssetStore writes,‚Äù and defines `ImageRequest`, but nowhere is the **Spark‚ÜíImmersion request surface** defined:

  * Does Spark ever directly propose `ImageRequest.prompt_context`?
  * Or does Lens deterministically derive ImageRequests from SceneCard / world state?
* Consequence: image generation either becomes ad-hoc (‚Äúwhatever the narrator says‚Äù) or disconnected from narration.

**SI-02 ‚Äî ‚ÄúNo canonical ‚ÄòImmersionPlan‚Äô bundle‚Äù**

* You have schemas for individual pieces (ImageRequest, VoicePersona, SceneAudioState), but no single **turn-level plan** that can be:

  * validated
  * replayed as non-authoritative
  * cached/attributed
* Consequence: the seam remains ‚Äúside-effecty glue code,‚Äù hard to reason about, hard to test.

**SI-03 ‚Äî ‚ÄúVoice persona selection protocol is incomplete‚Äù**

* VoicePersona exists; Lens voice resolver WO-052 is about mapping characters to personas.
* Missing at the seam: how Spark tags *who is speaking* in a way that deterministically maps to a persona:

  * speaker_id rules
  * multi-speaker narration formatting
  * fallbacks when speaker ambiguous
* Consequence: every character collapses to a default voice even if voice tech works (this matches WO-051‚Äôs stated problem).

**SI-04 ‚Äî ‚ÄúAttribution ledger integration point is not specified‚Äù**

* Immersion supports AttributionLedger, but the seam doesn‚Äôt specify:

  * when to write attribution (on generation? on cache hit?)
  * which component is responsible (Spark adapter? immersion adapter? orchestrator?)
* Consequence: provenance is inconsistently captured or silently skipped.

**SI-05 ‚Äî ‚ÄúNon-blocking rule vs quality gates (image critique, audio selection)‚Äù**

* Immersion must not block engine execution, but you also want quality gating (image critique pipeline is called out in project context / GO gates).
* Missing seam protocol: how to reconcile ‚Äúnon-blocking‚Äù with ‚Äúregenerate until acceptable‚Äù:

  * async regeneration?
  * bounded retries during prep only?
  * placeholder-first policy?
* Consequence: either immersion blocks (violating boundary) or quality gates are toothless.
