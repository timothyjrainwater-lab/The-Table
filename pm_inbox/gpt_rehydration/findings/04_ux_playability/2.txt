# AIDM Research Gap Register — Player Experience Focus

This analysis ignores engine correctness and focuses strictly on **what the player experiences**: first launch, first session, mid-session failures, recovery, and session continuity.

---

# Section 1 — Critical Path Player-Experience Blockers

---

## RG-PX-001 — Undefined First-Run Onboarding Flow

**Question:**
What exactly happens the first time a user launches AIDM? How do they:

* Select models?
* Validate hardware compatibility?
* Generate their first dungeon?
* Choose characters?
* Understand the “Prep vs Play” distinction?

**Source Context:**

* README defines prep/play split but does not define onboarding sequence or UI/CLI flow.
* Hardware baseline satisfied, but no first-run wizard exists.
* Spark + SDXL cannot load simultaneously — but no player-facing explanation or auto-handling spec exists.

**Blocks:**

* First playable session.
* MVP validation.
* Any non-developer user adoption.

**Research Type:** Design + Prototype
**Recommended Approach:**
Design a deterministic “First Session Script” covering:

1. Model detection.
2. VRAM validation.
3. Optional content generation.
4. First combat encounter.
5. First narration event.

Then simulate it as a scripted dry run.

**Scope:** Medium

---

**Blocks:**

| ----- | ------- | ----------- | ------------- | -------- |

* Retry silently
* Fall back to templates

# Section 5 — Implicit Assumptions Affecting Player Experience

---

The docs assume 55 templates are acceptable until Spark fully integrated.

Players will interpret this as low intelligence.

## Assumption 2 — Voice Is Optional

Voice pipeline is treated as immersion-only.

In practice:
Voice-first positioning makes voice a core promise.

Player perceives crash.

UX needs layered degradation.
