PurposeThis is a secondary-pass audit over the conversation to capture fine-grained details that could be lost between high-level specs.It consolidates: - micro-requirements - UX nuances - boundary conditions - implied implementation needsIt is intended to be used as a checklist during planning and ticket writing.1) Core Architectural Separation (Reconfirmed)1.1 Canonical vs PresentationCanonical IDs are the only truth for mechanics.Presentation (names, descriptions, assets, voice, language) is replaceable.Skinning/reskinning is allowed only at presentation level.1.2 “Possible vs Required” Scope BoundaryReskinning is an architectural property.Do not commit publicly to supporting all genres/languages.Keep “infinite flexibility” as internal capability, not a launch promise.2) Generative Asset Strategy (Fine Detail)2.1 Prep-First Asset GenerationMost generation happens during campaign/session prep, not during live play.Live generation may exist as an optional fallback, but must be bounded.2.2 Asset Logging & ReuseGenerated assets must be:archivedtaggedretrievablereusable across future sessionsGoal: minimize repeated generation and improve continuity.2.3 Continuity HandlingNPCs use static anchor images (generated once, reused).Scenes use background plates.Runtime presentation is primarily compositing (portrait/sprite over background).3) Image Generation + Critique Gate (Fine Detail)3.1 Hard Requirement: Critique/Quality GateImage generation must be paired with critique/quality evaluation.Generation without critique risks unusable outputs.3.2 Critique Should Be Checklists (Not Vibes)Quality evaluation should cover: - readability at intended UI size - subject centering/cropping - obvious artifacting (faces/hands/edges) - style adherence to campaign tone - match to anchor identity (for NPC portraits)3.3 Regeneration Loop ConstraintsBounded attempts (avoid infinite loops).Heuristics first, critic model second where needed.Output caching keyed to canonical IDs to prevent drift.4) Voice Strategy (Fine Detail)4.1 Priority: TTS > STT (Experience)TTS quality is more immersion-critical than STT accuracy.Output must be natural, expressive, controllable by AIDM.4.2 STT Trust SafeguardsUse confirmation flows for high-impact commands.Support repeat/repair interactions.Multilingual input must map through alias tables to canonical IDs.4.3 Voice as a Presentation PluginVoice profiles are importable/swappable.Voice changes do not affect mechanics.Voice must have text equivalents (accessibility).5) Sound Generation (Fine Detail)5.1 Sound Palette During PrepGenerate and tag a sound palette during prep:monster vocalizationsweapon impactsUI stingersambience loopsmusic themes (exploration/combat/tension)5.2 Runtime Uses Selection/MixingRuntime should primarily select/mix from palette.Goal: “produced” feel, not constant live generation.6) Memory / Logs / Indexing (Fine Detail)6.1 The LLM Should Query Truth, Not Hold ItTruth stored in indexed records; LLM retrieves.Minimize context-window stress.6.2 Memory Objects (Concrete)Maintain structured records for: - entity cards (NPCs/locations/factions) - relationships - event timeline - open threads/hooks - inventory and state - consequence explanations (e.g., alignment shifts)6.3 Session Recap as Proof of MemoryOn launch, DM can recap prior sessions.DM can reference specific moments (e.g., “three goblins; arrow hit”).DM can justify consequences using logs.7) Onboarding & Session Entry UX (Fine Detail)7.1 Onboarding is Gameplay (Session Zero)Avoid “click campaign” framing.The DM speaks first; onboarding is interactive.7.2 DM Persona Selection (First 30 Seconds)DM greets user.DM asks if user likes this DM style.User requests different persona.DM confirms switch (“How about now?”).Must be fast enough to feel responsive.7.3 Player Experience CalibrationDM may ask:“Are you new here or old-school?”“Do you want explanations or just results?”“Do you want to see dice rolls or keep it fast?”DM uses answers to tune pacing, verbosity, teaching level.7.4 Ceremony is ConsentDice and other ritual elements must be:skippableadjustablechosen conversationally7.5 Dice Animation Quality is CrucialDice are ritual: fairness + anticipation.Visual dice + satisfying audio.Must never misrepresent randomness.7.6 Dice Customization (Taste of Power)DM can ask preferences:big vs small dicecoloreffects (sparkles)Purpose: reveal generative capability early.Customization is expressive only; no mechanical impact.Persist choices; allow later changes in conversation.7.7 Character Creation as Guided ConversationDM rolls stats, shows results.Player assigns values by intent (“I want to be strong; put my 18 there”).DM explains stats in child-friendly, contextual terms.Explanations are optional and adaptive.7.8 Explicit Prep Contract & AnticipationDM sets expectation for prep time (e.g., ~1 hour).This is a clean handoff: user understands why and feels excited.The “taste of possibility” (voice/image/dice) reduces abandonment.8) Accessibility & Inclusive Design (Fine Detail)8.1 Text Parity is MandatoryText input/output must always be available.Supports users who are deaf, hard-of-hearing, non-speaking, or in constrained environments.8.2 No Voice-Only Lock-InVoice is primary, not required.Onboarding and gameplay must not assume audio.9) Daily Launch Interaction (Newly Emphasized)9.1 The Home Screen is a ConversationEach app launch begins with DM greeting.DM asks readiness; user can request recap/notes.DM can discuss sheet changes (e.g., alignment shift) and explain why.9.2 Avoid “Creepy Omniscience” ToneMemory should feel contextual and warm.Provide opt-outs (skip greeting/recap) if needed.10) Hardware & Optimization + image + critique.Plan GPU/CPU fallback paths.Avoid selecting models that assume rare hardware.11) Research Phase Requirements (Fine Detail)11.1 Must Research Before Lock-InTracks: - LLM - image gen
