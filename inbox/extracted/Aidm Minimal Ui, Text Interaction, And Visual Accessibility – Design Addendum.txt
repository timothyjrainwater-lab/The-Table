PurposeThis document captures and locks a previously understated but critical UI principle: minimal, unobtrusive text interaction and visual accessibility.It serves as an addendum to onboarding, accessibility, and persistent DM interaction specs.The goal is to ensure that: - voice-first design is preserved - text interaction is always available - accessibility needs are met - UI never competes with the DM for attentionCore PrincipleUI should be available, not demanding.The interface exists to support interaction, not to visually dominate it.Voice remains primary; text is a quiet, reliable fallback and supplement.Text Interaction Layer (Chat Window)PresenceA lightweight text interaction window must always be available.Placement should be conventional and non-intrusive (e.g., bottom corner, MMO-style).The window should not demand attention through animation, flashing, or color shifts.PurposeThe text window supports: - accessibility (deaf / hard-of-hearing / non-speaking users) - noisy or private environments - quick clarification or command entry - reviewing prior DM outputIt is not the primary UI.Clickable Text & Light InteractablesScopeCertain DM outputs may include optional clickable elements, such as: - spell names - item names - conditions - recap referencesThese act as assistive affordances, not required navigation.Design ConstraintsClickable text must not visually dominate (no heavy underlines, no bright colors).Hover or focus states should be subtle.Clickability is discoverable but not emphasized.If a user never clicks anything, the system must still be fully usable.Visual Accessibility RequirementsFont CustomizationThe system must support: - adjustable font size - readable default fonts - high legibility at small sizesColor AccessibilityText coloration must be safe for common forms of color blindness.Information must never be conveyed by color alone.High-contrast modes must be available without changing layout.Interaction PhilosophyNo Menu DependencyCore interactions must not require navigating settings menus.Preferences (e.g., text size, pace, verbosity) should be adjustable conversationally where possible.Progressive DiscoveryUI affordances should reveal themselves naturally through use.New users are not expected to understand all interactions immediately.Relationship to Player ModelingThe system may infer text vs. voice preference from usage patterns.Player profile may bias toward presenting more or less text detail.Explicit user choices always override inference.Non-GoalsThe UI is not a dashboard.The UI does not compete with voice narration.The UI does not surface complex configuration trees.StatusThis addendum locks minimal UI presence, text interaction, and visual accessibility as supporting systems, not competing interfaces.
