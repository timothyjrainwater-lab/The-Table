PurposeThis document formalizes and locks the architectural requirements, constraints, and implementation considerations that emerged across prior discussions. It is intended to be binding input for the Action Plan and Development Roadmap, not exploratory notes.The core intent is to ensure the system remains mechanically deterministic, presentation-agnostic, generatively skinnable, and open to localization and input extensibility without introducing scope creep or unstable commitments.Foundational PrincipleThe system ships physics, not props.Canonical mechanics (rules, numbers, legality, state transitions) are invariant.Fictional presentation (names, descriptions, visuals, audio, voice, language) is a replaceable, generative rendering layer.Generative AI operates strictly in the presentation layer and is never an authority over mechanics.This separation is non-negotiable.Canonical Core (Locked)1. Canonical IDsAll mechanically relevant entities must be defined by stable, language-agnostic identifiers:ItemsSpells / AbilitiesMonsters / NPC archetypesConditions / Status effectsActionsEventsRules engine, logs, validation, and replay operate exclusively on these IDs.Human-readable strings are never authoritative.2. Deterministic Engine ContractGiven identical inputs (player intent → canonical resolution), the engine must always produce identical outcomes.Randomness is seeded and logged.Event logs are replayable independent of presentation layer.Presentation changes must not affect: - legality of actions - numeric outcomes - state transitionsGenerative Presentation Layer (Locked)3. Skin Packs (Primary Abstraction)A Skin Pack is a data-driven + generative presentation bundle keyed to canonical IDs.A Skin Pack may provide: - Display names - Aliases (for voice and text input) - Descriptions (short / long) - Tone and stylistic constraints - Visual generation prompts or references - Audio / voice generation profilesA Skin Pack may not: - Alter mechanics - Add modifiers - Introduce new action legalitySkin Packs are declarative, validated, and hot-swappable at the presentation level.4. Generative Renaming & RephrasingExamples: - weapon.hand_crossbow → “Laser Blaster” - spell.fireball → “Plasma Detonation”Constraints: - Canonical ID remains unchanged - Damage type, range, area, scaling remain unchanged - AI generation is constrained by locked semantic metadata (function, category, tone)Terminology locking is required to prevent synonym drift mid-session.Localization & Language (Locked)5. Output LocalizationAll player-facing text is rendered via templates keyed to canonical IDs.Localization applies only to presentation strings and templates.Grammar-safe placeholders are mandatory (e.g. {item_name}, {damage_value}).Languages supported are not hard-coded. Language packs are importable.6. Multi-Language INPUT (Explicit Requirement)The system must accept player input (text and voice) in multiple languages.Requirements: - Input is resolved via alias tables mapped to canonical IDs - Aliases are language-specific and Skin Pack–specific - Canonical engine never processes localized stringsAmbiguity handling and disambiguation must be explicit.This is a design requirement, not a post-launch feature.Voice & Audio (Locked Boundaries)7. Voice Profiles as Presentation PluginsVoice generation parameters (accent, timbre, pacing, language) are treated as presentation plugins.They may be imported or swapped without affecting: - mechanics - intent resolution - game stateVoice plugins must not redefine meaning—only delivery.Import & Extensibility Model (Locked)8. Importable Content TypesThe system must support importing: - Adventures (event graphs, encounters, states) - Skin Packs - Language Packs - Voice / Tone PacksAll imports are validated against strict schemas.9. Import Safety RulesImported content may: - decorate - localize - narrateImported content may not: - redefine mechanics - override canonical IDs - introduce unvalidated aliasesAny violation fails import.Explicit Non-Goals (Locked)The system does not promise: - Support for all genres - Mechanical parity across genres - Automatic fulfillment of genre-implied mechanics (e.g. hacking, netrunning)Reskinning is cosmetic unless explicitly extended later.Risk Controls (Required)10.
