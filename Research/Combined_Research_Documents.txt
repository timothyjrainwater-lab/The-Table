# Action Plan Addendum — CP-14–CP-20 Determinism Constraints
Status: Binding once merged
Purpose: This addendum formalizes determinism-critical constraints identified in the PHB/DMG audit so future instruction packets (CP-14–CP-20) do not implicitly violate replayability, event ordering, or scope discipline.
## 1. Locked Deferrals (Non-Negotiable)
### 1.1 Readied & Delay Actions
Readied actions (PHB p.160) are explicitly deferred beyond CP-20.
Rationale: interrupt-driven execution introduces nested event ordering and natural-language triggers incompatible with current deterministic guarantees.
Allowed: static analysis notes and schema placeholders only.
### 1.2 Grapple System Scope
Grapple is limited to a simplified model in CP-18:
Apply grappled condition only (movement denied; limited actions).
Pinned, escape sequences, and grapple damage loops are deferred.
Rationale: full grapple creates state explosion and complex interrupt windows.
## 2. Determinism Requirements (Must Hold)
### 2.1 Diagonal Movement
Diagonal movement (PHB p.148) must record path history to enforce RAW 1-2-1-2 cost.
No Euclidean or Manhattan shortcuts are permitted.
Movement intents must carry path_taken (ordered grid points).
### 2.2 Interrupt Ordering
No mid-action interrupts are allowed prior to CP-15.
AoO (CP-15) must resolve as:
Trigger emission
All legal AoOs resolved
Original action completes or aborts
## 3. Critical Path Confirmation
The only critical path for playable combat remains: - CP-14: Initiative & Action Economy - CP-15: Attacks of Opportunity - CP-16: Conditions & Status Effects - CP-17: Saving Throws
No packet may skip or collapse this order.
## 4. Documentation Consistency Rule
Any future packet that touches: - Initiative order - Movement costs - Interrupts or reactions - Condition mechanics
Must reference this addendum explicitly in its acceptance section.
## 5. Scope Integrity
This addendum does not: - Expand runtime scope - Introduce new systems - Authorize LLM decision-making
It only locks constraints already assumed by audits and plans.
Effective Upon Merge

# Action Plan Amendment — Cross‑Cutting Mechanics Coverage & Mounted Combat
Status: Binding once merged
Applies To: All instruction packets CP‑17 and onward
## 1. Purpose
This amendment addresses a discovered structural risk: the existence of cross‑cutting RAW subsystems (e.g., mounted combat) that cannot be safely implemented as emergent behavior from isolated kernels (movement, attacks, saves, conditions).
The amendment introduces: - An explicit governance mechanism to prevent silent omissions - A mandatory audit process to surface similar subsystems - A correction to the core action plan to include Mounted Combat as a first‑class packet
## 2. Rules Coverage Ledger (New Required Artifact)
### 2.1 Definition
A Rules Coverage Ledger (RCL) is a canonical, append‑only project artifact that enumerates all RAW mechanical subsystems relevant to runtime play.
Each entry MUST include: - Subsystem name - Source reference (PHB / DMG / MM section) - Classification: atomic | cross‑cutting | dm‑discretion - Coverage status: implemented | partial | deferred | out_of_scope - Owning packet (CP‑## or amendment) - Determinism risk: low | medium | high - Notes (edge cases, propagation rules, known hazards)
### 2.2 Governance Rule
No packet may be frozen unless: - All subsystems it touches are present in the RCL - Each such subsystem has an explicit coverage status
Silence is forbidden. Every subsystem must have a declared disposition.
## 3. Mandatory Cross‑Cutting Mechanics Audit
### 3.1 Audit Trigger
A Cross‑Cutting Mechanics Audit (CCMA) MUST be performed when: - A new interaction class is introduced, or - A packet alters assumptions about actor identity, position, or action ownership
### 3.2 Audit Scope
The audit MUST examine PHB, DMG, and MM mechanics to identify subsystems that: - Span two or more kernels (e.g., movement + AoO + saves) - Introduce composite or coupled actors - Cause bidirectional state propagation
### 3.3 Required Outputs
The audit MUST produce: - An updated Rules Coverage Ledger - A Gap Register listing uncovered or mis‑sequenced subsystems - A sequencing proposal for any newly discovered packets
## 4. Action Plan Correction — Mounted Combat
### 4.1 Finding
Mounted combat is a RAW‑critical, cross‑cutting subsystem that: - Couples two independent entities (rider and mount) - Rewrites movement, AoO, attack geometry, saves, and condition propagation
It cannot emerge safely from existing kernels.
### 4.2 New Packet (Inserted)
CP‑18A — Mounted Combat & Rider–Mount Coupling
Prerequisites: - CP‑14 Initiative & Action Economy - CP‑15 Attacks of Opportunity - CP‑16 Conditions - CP‑17 Saving Throws
Scope (Non‑Negotiable): - Rider–mount coupling state - Controlled vs independent mounts - Shared movement and AoO routing - Mounted attack modifiers and charge rules - Save and condition propagation (mount ↔ rider) - Dismount and fall resolution
Explicit Non‑Goals: - Exotic mounts - Mounted spellcasting edge cases - Ride skill optimization beyond legality checks
### 4.3 Action Plan Ordering
The critical path is amended as follows:
CP‑14 → CP‑15 → CP‑16 → CP‑17 → CP‑18A (Mounted Combat) → CP‑18 (Combat Maneuvers)
No packet may bypass CP‑18A if it depends on mounted interactions.
## 5. Enforcement
The Rules Coverage Ledger is a blocking artifact for packet freeze
CCMA findings are binding once merged
Any discovered omission MUST be resolved via a new packet or explicit deferral
## 6. Summary
This amendment eliminates the failure mode of silent system omission by: - Forcing explicit coverage declarations - Elevating cross‑cutting mechanics to first‑class design concerns - Correcting the action plan to include Mounted Combat as required infrastructure

# Action Plan Amendment — High-Risk Cross-Cutting Combat Subsystems
Status: Binding once merged
Applies To: Core action plan, Rules Coverage Ledger, and all instruction packets CP-17 and onward
## 1. Purpose
This amendment incorporates findings from the Cross-Cutting Mechanics Audit and addresses a second class of structural risk beyond mounted combat: high‑risk combat subsystems that silently violate core engine assumptions if treated as emergent or modifier‑only behavior.
The goal is to ensure that all such subsystems are: - Explicitly identified - Tracked in the Rules Coverage Ledger (RCL) - Either scheduled as first‑class packets or explicitly deferred
No RAW subsystem that alters targeting legality, spatial reasoning, or actor identity may remain implicit.
## 2. Definition: High‑Risk Cross‑Cutting Subsystem
A subsystem is classified as High‑Risk Cross‑Cutting if it meets any of the following criteria:
Couples multiple entities into a composite or dependent state
Alters who can target whom, or whether targeting is legal at all
Changes threatened areas or reaction (AoO) geometry
Introduces epistemic state (who is observed, hidden, or detectable)
Requires bidirectional propagation of damage, saves, or conditions
Breaks the assumption of 2D ground‑based movement
Such subsystems MUST NOT be implemented as: - Pure numeric modifiers - Narrative‑only rulings - Side effects hidden inside other resolvers
## 3. Newly Identified High‑Risk Subsystems
The following subsystems are formally designated High‑Risk Cross‑Cutting and MUST appear in the Rules Coverage Ledger with explicit disposition.
### 3.1 Flight & Three‑Dimensional Movement
RAW Characteristics: - Altitude and vertical positioning - Ascent/descent limits - Falling and stall behavior - AoO geometry in three dimensions - Prone, helpless, or stunned interactions while airborne
Risk: Breaks 2D positioning, gravity assumptions, and AoO resolution.
Disposition: Not implemented. Requires a dedicated packet before any broad environmental expansion.
### 3.2 Cover, Concealment, and Miss Chance
RAW Characteristics: - Partial vs total cover - Concealment percentages - Total concealment (illegal targeting) - Soft cover via creatures - Interactions with AoOs, sneak attack, and touch attacks
Risk: Affects attack legality, not just attack math; intersects perception and positioning.
Disposition: Not implemented. Must be modeled explicitly as a visibility/targeting kernel.
### 3.3 Invisibility and Observed vs Unobserved State
RAW Characteristics: - Invisible but audible creatures - Flat‑footed interactions - Loss of invisibility on attack - AoO restrictions - Targeting failures without precise location
Risk: Introduces epistemic state and breaks the assumption that all entities are mutually observable.
Disposition: Not implemented. Requires explicit observation state tracking.
### 3.4 Charge (Full RAW Semantics)
RAW Characteristics: - Straight‑line movement constraints - AoO exposure - Attack bonuses and AC penalties - Weapon‑specific interactions (e.g., lance)
Risk: Charge is not equivalent to “move + attack” and will misresolve if treated as such.
Disposition: Partially represented. Full RAW behavior must be isolated into a dedicated resolver or packet.
### 3.5 Underwater Combat
RAW Characteristics: - Movement mode replacement - Weapon effectiveness changes - Spell and fire restrictions - Drowning and suffocation rules
Risk: Cross‑cuts movement, attacks, saves, and conditions; incompatible with land‑based assumptions.
Disposition: Not implemented. Explicit deferral required.
### 3.6 Falling and Forced Movement
RAW Characteristics: - Damage scaling by distance - Forced movement into hazards - Trip/bull rush leading to falls - Mounted and flying edge cases
Risk: Introduces non‑voluntary movement and delayed damage resolution.
Disposition: Not implemented. Must not be handled narratively.
## 4. Action Plan Requirements
### 4.1 Mandatory Ledger Entries
All subsystems listed in Section 3 MUST: - Appear in the Rules Coverage Ledger - Be marked deferred or planned - Identify expected owning packets
Silence or implicit assumption is forbidden.
### 4.2 Sequencing Rule
No packet may: - Introduce partial behavior for a High‑Risk subsystem, or - Depend on such a subsystem
unless that subsystem has either: - A completed owning packet, or - An explicit deferral with enforced non‑use
## 5. Enforcement & Audit
PBHAs MUST verify that no High‑Risk subsystem has leaked partial behavior
Any discovered leak constitutes a blocking defect
Corrections MUST occur via new packets or amendments, not ad‑hoc fixes
## 6. Summary
This amendment expands project governance from “what we are building next” to “what we are explicitly choosing not to build yet.”
By classifying and tracking High‑Risk Cross‑Cutting subsystems, the project prevents a second mounted‑combat‑class failure mode and preserves determinism, correctness, and auditability as scope increases.

# Action Plan Update — Adventure Ingestion, Assistant Mode, and Play Lanes
## Status
Binding once merged. This document updates the project action plan to explicitly incorporate: - Adventure ingestion from OCR/PDF sources - Autonomous rulings during ingestion - DM assistant mode - Multiple play lanes sharing a single deterministic runtime
This update is written to be consistent with existing governance documents (Project Coherence Doctrine, Tooling & Interfaces Contract).
## 1. Clarified Design Intent
### 1.1 Human‑DM‑Analog Ingestion Philosophy
During ingestion, the system is explicitly permitted to behave like a human DM preparing an adventure: - It may interpret incomplete, ambiguous, or contradictory source material. - It may invent details where the source is silent. - It may resolve conflicts using internal heuristics rather than user confirmation.
Ingestion rulings are allowed to be subjective.
However: - These rulings are prep‑time only. - Every ruling must be explicitly recorded. - Runtime determinism is preserved by treating ingestion output as authored content, not live interpretation.
This mirrors how real DMs work: interpretation happens before play, not invisibly during resolution.
## 2. Two‑Phase Adventure Ingestion Model
### 2.1 Phase A — Ingestion & Compilation (Non‑Authoritative)
Inputs - OCR PDF of an adventure module - Optional user configuration (fidelity level, tone, danger level)
Allowed Behaviors - Natural‑language interpretation of prose - Extraction of encounters, locations, NPCs, and hooks - Invention of missing mechanical details (DCs, monster counts, treasure placement) - Re‑sequencing or restructuring scenes for internal coherence
Ruling Policy - If a “true RAW answer” cannot be derived, the system must make a ruling. - The system must not defer endlessly or request user arbitration by default.
Outputs - A compiled Module Pack (structured, versioned, deterministic) - A Rulings Ledger recording: - Source excerpt reference (page/section when available) - Nature of ambiguity - Chosen ruling - Rationale
The Module Pack becomes the authoritative artifact for runtime.
### 2.2 Phase B — Runtime Execution (Authoritative & Deterministic)
At runtime: - The engine does not reinterpret the PDF. - The engine does not invent new rules or rulings. - The engine executes the Module Pack exactly like any other authored scenario.
Any further ambiguity must be handled by: - Preconfigured runtime policies, or - Explicit, logged runtime rulings (rare, surfaced clearly)
## 3. Module Pack Definition (v0)
A Module Pack is treated as authored campaign content.
Minimum Required Sections - Metadata (title, source, ingestion timestamp, version) - Global assumptions (tone, lethality, interpretation policies) - Scene graph (locations, transitions) - Encounters (monsters, tactics hints, triggers) - NPCs (stats or references, motivations) - Treasure & rewards - Rulings Ledger (append‑only)
Module Packs are immutable once runtime begins.
## 4. Play Lanes (Unified Runtime)
The system supports three play lanes that differ only in content source:
Sandbox Lane
Procedurally generated content
Deterministic seeds + world facts
Scenario‑Setting Lane
Manually authored Module Packs
System‑generated or human‑generated
Adventure‑Book Lane
Module Packs produced via OCR/PDF ingestion
All lanes share: - The same deterministic runtime - The same combat engine - The same event log and replay guarantees
## 5. DM Assistant Mode
### 5.1 Purpose
DM Assistant Mode exists to support human‑run games or module preparation, not to replace the deterministic engine.
### 5.2 Capabilities
Rules lookup with citation
Outcome simulation (“What if this encounter runs?”)
Ingestion review (inspect rulings, edit Module Packs)
Non‑binding tactical advice
### 5.3 Restrictions
Assistant mode cannot mutate canonical WorldState without explicit command
Assistant mode cannot silently apply rulings during runtime
## 6. Encounter Sandbox & Simulation
Encounter simulation is promoted as a first‑class feature.
Inputs - Party composition - Monster composition - Environment seed - Tactics policies
Outputs - Deterministic event logs - Outcome distributions (via repeated deterministic runs with controlled variation) - Human‑readable transcripts
This feature doubles as: - A DM tool - A regression test harness
## 7. Updated Development Phases
### Phase 0 — Vertical Slice (Unchanged, Non‑Negotiable)
One scene
One combat
Deterministic replay
### Phase 1 — Encounter Sandbox
ScenarioRunner
Tactics policies
Transcript output
### Phase 2 — Assistant Mode
Interaction mode switch
Rules citation system
### Phase 3 — Module Pack v0
Schema definition
Manual authoring support
### Phase 4 — OCR/PDF Ingestion Compiler
Best‑effort extraction
Autonomous rulings
Rulings Ledger output
### Phase 5 — Lane Selection UX
sandbox | scenario | module
## 8. Non‑Goals (Reaffirmed)
No live PDF interpretation during runtime
No hidden or unlogged rulings
No LLM authority over mechanical resolution
## 9. Summary
This update explicitly allows the system to: - Treat adventure books as inspiration plus constraints, not sacred text - Make autonomous preparatory rulings like a human DM - Preserve determinism and replayability during play
This resolves the tension between creative freedom and system integrity by moving interpretation into a logged, prep‑time compilation phase.

# Action Plan Update — Alignment & Campaign Memory Foundations
Status: Action Plan Amendment
Purpose: This document updates the project action plan to formally include Alignment, Divine Influence, and Campaign-Scale Memory as planned future work. It defines scope boundaries, sequencing, and foundational rules so implementation can proceed later without architectural conflict or scope ambiguity.
## 1. Scope Clarification
### 1.1 Clarified Boundary
The project remains out of scope for campaign planning tools, UI, or authoring workflows.
However, the following are now explicitly in scope as foundational systems: - Campaign continuity records - Long-term character behavior evaluation - Alignment drift tracking - Divine and religious consequence modeling
These systems support narrative coherence and rule fidelity but do not prescribe story outcomes.
## 2. Guiding Principles (Foundation Rules)
Record First, Judge Later
All alignment- and memory-related data is recorded descriptively before any evaluation occurs.
Gradualism Over Absolutes
Alignment and belief changes occur only through sustained behavioral patterns.
Transparency of Consequence
Any surfaced alignment or divine consequence must be explainable with cited evidence.
Player Freedom Preserved
No action is blocked by alignment systems.
Alignment influences world response, not permissions.
Narrative-Led, Mechanically-Grounded
Narrative signals precede mechanical consequences wherever possible.
## 3. Roadmap Integration
### Phase A — Campaign Memory & Alignment Records (Schema-Only)
Goal: Establish durable, deterministic data structures.
Deliverables: - Alignment Evidence Record schema - Session-linked evidence entries - Immutable linkage to session ledgers - Validation rules (fail-closed) - Full test coverage
Non-Goals: - No evaluation logic - No alignment shifting
### Phase B — Alignment Drift Evaluation (Analytical Layer)
Goal: Introduce slow, explainable alignment evaluation.
Deliverables: - Periodic evaluation process (session-windowed) - Trend detection rules - Threshold definitions - Player-facing explanation format
Constraints: - No real-time evaluation - No single-event flips
### Phase C — Divine Influence & Religious Consequences
Goal: Make gods and faith mechanically and narratively relevant.
Deliverables: - Deity value and alignment expectation data - Favor / disfavor tracking - Narrative signaling rules - Mechanical consequence hooks (RAW-aligned, citation-backed)
Constraints: - Behavior-based evaluation only - No punitive framing
## 4. Dependencies & Ordering
Phase A depends on existing Session Ledger and Event Log systems.
Phase B depends on Phase A.
Phase C depends on Phase B and existing class/religion rules.
No phase introduces combat or resolution logic.
## 5. Risk Management & Safeguards
Identified Risks and Mitigations:
Moral Fiat Risk: Mitigated by evidence logs and surfaced explanations.
Over-Determinism: Mitigated by trend-based evaluation and delayed consequences.
Narrative Sterility: Mitigated by prioritizing tone and implication over hard effects.
## 6. Impact on Future Development
This update ensures that: - Alignment systems are planned, not retrofitted. - Divine relevance is treated as systemic, not cosmetic. - Long campaigns remain coherent without limiting player agency.
## 7. Summary
Alignment and campaign memory are now formally recognized as foundational systems. This action plan amendment provides clear sequencing and boundaries, ensuring these systems can be implemented later without architectural rework or scope confusion.

## Phase 3.6 — Combat Engine Specification
### Purpose
Define combat as a deterministic, auditable execution contract layered on top of the core Rules Engine. This phase formalizes how combat state is tracked, how turns resolve, and how mathematical operators are applied in strict order, ensuring absolute correctness without exposing video‑game abstractions (e.g., HP bars) to players.
### 3.6.1 Combat as a System
Formal definition:
Combat = (WorldState₀, InitiativeOrder, TurnLoop, ResolutionOperators, EndConditions)
Where: - WorldState₀: Snapshot of all entities, terrain, effects, and hidden information at combat start. - InitiativeOrder: Ordered list of combatants, mutable only by ready/delay effects. - TurnLoop: Deterministic per‑round execution cycle. - ResolutionOperators: Pure mathematical functions (attack, save, opposed check, etc.). - EndConditions: Explicit triggers that terminate combat.
Combat is a bounded execution context with stricter time and action constraints than normal play.
### 3.6.2 Combat State Model
Combat state is partitioned into four mandatory layers.
#### A) Spatial State
Position (grid or abstracted distance)
Size category and occupied space
Reach and threatened squares
Movement modes (land, fly, swim, burrow, climb)
Terrain interaction (difficult terrain, elevation, squeezing)
Line of Sight (LoS) and Line of Effect (LoE)
Cover and concealment categories
#### B) Temporal State
Initiative order
Current round count
Start‑of‑turn and end‑of‑turn hooks
Duration tracking (rounds, minutes, hours)
#### C) Entity State (DM‑canonical)
HP (current, max, temporary) — exact values DM‑only
Armor Class breakdown (normal, touch, flat‑footed)
Saving throws (Fort, Ref, Will totals)
Attack routines and damage expressions
Conditions and status flags
Action economy usage (standard, move, full, swift, immediate, AoOs)
Resources (spell slots, charges, rage rounds, etc.)
#### D) Information State
Public vs private information boundaries
Hidden creatures, traps, and zones
Uncertainty gates (concealment, miss chance, SR)
### 3.6.3 Canonical Combat Algorithm

## Phase 3.7 — DM Decision Policy Engine (Combat Choice Model)
### Purpose
Implement the non-rules part of combat DMing: how enemies (and encounter-level pacing systems) choose actions while remaining: - Rules-legal (all options validated by the Rules Engine) - Fiction-consistent (monster doctrine/personality) - Fun/fair (agency + telegraphing + guardrails)
This is a policy layer that sits above the deterministic Rules Engine. It never changes math; it only selects among legal actions and shapes presentation.
### 3.7.1 Inputs (Feature Set)
At each decision point, the policy receives a structured feature bundle:
#### A) Encounter-Level Features
Encounter purpose (tutorial/resource tax/boss/chase/puzzle-combat/narrative beat)
Win conditions (kill/delay/escape/protect/disrupt/hold)
Target rounds (expected length)
Reinforcement triggers/timers
Environment topology (terrain channels, hazards, visibility)
#### B) Side-Level Features
Action economy (effective actions/round after control)
Threat distribution (who is dealing damage/control)
Resource posture (observed slots/charges/HP bands)
Morale posture (disciplined/cowardly/fanatical/cunning)
#### C) Per-PC Inferred Features (Belief-Based)
Maintain belief distributions (not ground truth) over: - Defense profile (AC/touch/ff bands; save bands) - Vulnerability flags (low Will/Ref; immobility; lack of ranged) - Offense profile (DPR band; control potency) - Mobility profile (flight/teleport/escape tools) - Role signature (striker/controller/support/tank/scout)
#### D) Per-Monster Features
Doctrine / goal (protect leader, kill healer, delay, steal, escape)
Burst potential and recharge/limited-use tracking
Control suite (stun/grapple/fear/trip)
Survivability (AC/DR/ER/SR + escape tools)
Position leverage (reach/AoOs/flight)
#### E) Local Tactical Features
Distances + reachable squares
Threatened squares + AoO risk
LoS/LoE availability
Cover/concealment vs key attackers
Flank opportunities and choke points
Buff/debuff timers (remaining rounds)
### 3.7.2 Objectives (Utility Function)
Represent DM choice as optimizing a weighted utility:
U = w₁·Danger + w₂·Agency + w₃·Pace + w₄·RulesIntegrity + w₅·FictionIntegrity
Weights are set by encounter purpose and monster doctrine.
#### Quantifiable Proxies
Danger: PC drop probability (1–2 rounds), TPK risk (bounded), resource burn rate, action denial risk
Agency: counterplay availability, telegraphing strength, outcome sensitivity to player choices
Pace: decision density, swing moments frequency, resolution latency (avoid complex stalls)
### 3.7.3 Candidate Generation (Legal Action Enumerator)
Given the current state, generate only rules-legal candidates: - Damage line (focus fire / burst) - Control line (deny key actions, reposition via forced movement) - Reposition line (cover, break LoE, set flanks, choke control) - Objective line (interact/escape/trigger hazard) - Defensive line (withdraw, total defense, heal, buff) - Exit line (flee/surrender/bargain/hostage) when morale triggers
All candidates must be validated by the combat legality checker before scoring.
### 3.7.4 Scoring (Per-Action Evaluation)
For each candidate action a, compute: - P_success(a) using discrete d20 math (exact counts, not approximations) - EV_damage(a) or EV_effect(a) including mitigation and miss gates - ΔActionEconomy(a) (expected actions denied / preserved next round) - ΔPositionLeverage(a) (cover, flanks, AoO advantage, LoE denial) - ΔSurvival(a) (monster survival probability change) - Counterplay(a) (does this leave meaningful responses?) - Complexity(a) (penalize slow adjudication lines) - FictionFit(a) (doctrine/intelligence/morale constraints)
Example composite:
Score(a) = α·EV(a) + β·Leverage + γ·Survival − δ·Unfairness − ε·Complexity
#### Probability Notes (Engine Contract)
Attack rolls: nat 1/20 bounds apply
Saves: no auto fail/succeed in RAW; compute discretely
Crits: P_crit = P_threat × P_confirm (apply concealment contract consistently)
### 3.7.5 Selection Policy (Avoid Robot Predictability)
Choose actions via controlled stochastic selection: - Argmax when clearly dominant - Softmax over top-k when close (temperature/“spice” parameter) - Personality affects temperature and aggression/caution balance
This produces variety without breaking fairness.
### 3.7.6 Belief & Uncertainty Update Loop
The policy must operate under incomplete information: - Unknown HP (by design) - Unknown saves/DR/ER/SR initially
Maintain belief distributions and update them after observed outcomes: - If fire spell does little → raise belief in fire resistance - If Will save repeatedly fails → lower Will band belief
Beliefs influence future scoring but never override RAW legality.
### 3.7.7 Guardrails (Social Contract + Fairness Telemetry)
Hard and soft constraints that prevent “gotcha DM” behavior: - Avoid repeated hard-locks without counterplay unless encounter is explicitly apex - Prefer telegraphed lethality (1–2 beats early) over surprise spikes - Ease-off triggers (example): two PCs down early with no counterplay, extreme crit variance
Mechanisms for easing while staying fiction-consistent: - Morale breaks / tactical retreat - Target switching consistent with monster intelligence - Overconfidence / misplays that still make sense
### 3.7.8 DM Policy API (Implementation Interface)
At each decision point: - Input: - state (combat state) - intent (encounter + monster goals) - options (validated legal actions) - Output: - chosen_action - score_breakdown (danger/agency/pace/rules/fiction + key terms) - telegraph_hints (what to narrate to maintain fairness)
### Phase Output
A deterministic, auditable combat choice policy that: - selects monster actions from legal sets, - preserves player agency and pacing, - remains fiction-consistent, - provides telegraph hooks for narration, - and integrates cleanly with the Rules Engine and Combat Execution Contract.

Phase 3.6 — Combat Engine Specification
Purpose
Define combat as a deterministic, auditable execution contract layered on top of the core Rules Engine. This phase formalizes how combat state is tracked, how turns resolve, and how mathematical operators are applied in strict order, ensuring absolute correctness without exposing video‑game abstractions (e.g., HP bars) to players.
3.6.1 Combat as a System
Formal definition:
Combat = (WorldState₀, InitiativeOrder, TurnLoop, ResolutionOperators, EndConditions)
Where: - WorldState₀: Snapshot of all entities, terrain, effects, and hidden information at combat start. - InitiativeOrder: Ordered list of combatants, mutable only by ready/delay effects. - TurnLoop: Deterministic per‑round execution cycle. - ResolutionOperators: Pure mathematical functions (attack, save, opposed check, etc.). - EndConditions: Explicit triggers that terminate combat.
Combat is a bounded execution context with stricter time and action constraints than normal play.
3.6.2 Combat State Model
Combat state is partitioned into four mandatory layers.
A) Spatial State
Position (grid or abstracted distance)
Size category and occupied space
Reach and threatened squares
Movement modes (land, fly, swim, burrow, climb)
Terrain interaction (difficult terrain, elevation, squeezing)
Line of Sight (LoS) and Line of Effect (LoE)
Cover and concealment categories
B) Temporal State
Initiative order
Current round count
Start‑of‑turn and end‑of‑turn hooks
Duration tracking (rounds, minutes, hours)
C) Entity State (DM‑canonical)
HP (current, max, temporary) — exact values DM‑only
Armor Class breakdown (normal, touch, flat‑footed)
Saving throws (Fort, Ref, Will totals)
Attack routines and damage expressions
Conditions and status flags
Action economy usage (standard, move, full, swift, immediate, AoOs)
Resources (spell slots, charges, rage rounds, etc.)
D) Information State
Public vs private information boundaries
Hidden creatures, traps, and zones
Uncertainty gates (concealment, miss chance, SR)
3.6.3 Canonical Combat Algorithm
Combat Start
Establish initial positions.
Determine surprise and awareness per side.
Resolve surprise round (standard or move only).
Roll initiative for all combatants.
Lock initiative order.
Per Round Loop
Repeat until an EndCondition is met.
Per Turn (Strict Order)
Apply start‑of‑turn effects.
Determine legal actions (conditions, restrictions).
Select action package (standard/move/full/etc.).
Resolve movement (costs, legality, AoO triggers).
Resolve primary action (attack, spell, maneuver, skill).
Apply secondary gates (SR, concealment, crit confirmation, immunities).
Apply damage or effects.
Update state (HP, conditions, positions, resources).
Apply end‑of‑turn effects.
Decrement durations.
3.6.4 Resolution Ordering Contract
All combat effects must resolve in the following order: 1. Target legality (range, LoS, LoE, type). 2. Primary roll (attack, save, or opposed check). 3. Immunity checks. 4. Concealment or miss chance. 5. Spell Resistance (if applicable). 6. Damage or effect calculation. 7. Mitigation (DR, ER, hardness). 8. Reaction triggers (AoOs, readied actions, immediate effects). 9. State update and duration ticking.
3.6.5 Mathematical Operator Set
Core operators: - d20 + modifiers ≥ target - Damage dice + static bonuses - Multiplier application (crit rules) - Mitigation application (DR, ER, immunity) - Opposed checks - Miss chance checks - Spell Resistance gates
All combat math is expressed as compositions of these operators.
3.6.6 Action Economy Enforcement
The engine must explicitly enforce: - Action slot availability per turn - AoO limits (including Combat Reflexes) - Swift and immediate action constraints - Ready and delay initiative mutations - Five‑foot step restrictions
Illegal actions must be rejected before resolution.
3.6.7 Conditions as Rule Modifiers
Conditions are not labels; each condition is a bundle of rule atoms that: - Modify attack rolls, AC, saves, speed, actions, or visibility - Impose or remove action permissions - Define duration and removal triggers
Condition application and removal occur only through state updates.
3.6.8 HP Presentation Policy
Exact HP values are tracked internally only.
Player‑facing output uses descriptive health states (e.g., fresh, wounded, staggering).
No health bars or numeric HP are exposed to players.
3.6.9 End Conditions
Combat terminates when one or more of the following is met: - All hostile entities neutralized - Surrender or flight - Objective achieved - Total party defeat or escape
Post‑combat cleanup (stabilization, resource accounting) follows core rule procedures.
Phase Output
A formally specified combat execution contract that integrates cleanly with the Rules Engine, supports voice‑driven DM adjudication, and guarantees mathematically correct outcomes under D&D 3.5 RAW.

Recreate the feeling of a real tabletop D&D session: - Voice-first interaction - Minimal UI - Strong narrative presence - Deterministic, rules-accurate mechanics - Transparent third‑party rule citations to prevent rules-lawyering

Outputs of this phase: - Campaign generation - Session breakdowns - NPC creation - Encounter design - Token & portrait generation - Rule citation anchors - Scene, NPC, and encounter “cards” for runtime use
This phase is creative, computationally heavy, and not latency-sensitive.
A readiness certificate confirms: - All assets exist - All encounters are rules-legal - All required citations are available - No missing dependencies
Only once certified can a session be played.
### Phase 2 — Play (Real-Time)
Once play begins: - No heavy generation - No context thrashing - Low-latency voice loop
The system loads only: - Current scene card - Relevant NPC cards - Active combat state - Rule citations on demand
This keeps the AI responsive and stable.

Character sheets (read-only + auto-updating)

Ability click-to-cast buttons
Inventory shopping lists
MMO-style cooldown bars
HP bars visible to players

## Combat & Targeting
Tokens are visually simple (colors or numbers)

Spells per day

Conditions

# AI-DM Future Systems (NON-BINDING)
Status: Working notes / design intent. Nothing here is implementation-law until promoted into the LOCKED spec.
## 1. Procedural Combat Field Generation (Default)
### 1.1 Goals (All Three)
Combat map generation should simultaneously optimize for: - Realism (geography, plausible structures) - Tactical richness (cover, choke points, flank routes, elevation) - Narrative flavor (scene identity, story-relevant props)
### 1.2 Reuse & Persistence
Reuse is allowed and expected when revisiting known locations.
Persistent locations (cities, forts, key dungeons) should have stored physical layouts.
Maps may mutate over time due to events (fires, collapses, construction).
### 1.3 Deterministic Under the Hood
Generation is deterministic given seed + world facts + prior mutations.
Players should not know seeds; determinism is for replay/test integrity.
## 2. Environment Mechanics (Solidity, Terrain, Traversal)
### 2.1 Solidity
Objects like trees, boulders, walls are solid: - block movement (unless special movement mode) - can block LoS/LoE
### 2.2 Terrain Types
Terrain should be tagged with mechanical effects (examples): - difficult_terrain (mud, rubble) - slippery (ice) - shallow_water, deep_water - dense_forest (visibility + movement)
### 2.3 Traversal Checks
Creative traversal (climb trees/walls, cross ice, swim) should invoke RAW skill checks with DCs.
### 2.4 DM Judgment DCs (RAW-Compatible)
RAW tables provide anchor examples.
DCs can be adjusted by contextual modifiers (snow, rain, moss, slope).
Chosen DCs must be logged for determinism/replay.
### 2.5 Player Context Injection
Before rolling, player may add relevant context via interrupt (e.g., “I have a climbing kit”). - Engine validates and incorporates into DC/bonuses per RAW. - No retroactive changes after the roll.
## 3. Plausibility Gate (Rulings Without Mercy)
### 3.1 Principle
Players may attempt creative actions, but some attempts are implausible in the fiction or illegal by RAW.
### 3.2 Required Behavior
System must be able to respond: - Yes (legal → resolve) - Requires clarification (missing state) - No (implausible/illegal) with explicit justification
### 3.3 Justification Format
When rejecting, provide: - Reason category: illegal_by_raw | physically_blocked | implausible_fiction - Key facts: “wall is smooth,” “no handholds,” “target not in LoS,” etc. - Optional citation if rules-driven
## 4. Time Model (Combat vs Narrative vs Exploration)
### 4.1 Combat Time
Combat rounds are 6 seconds (RAW).
### 4.2 Narrative Time
DM adjudicates conversational and scene time (e.g., “20 minutes talking to the shopkeeper”).
Time passage must be tracked and logged.
### 4.3 Exploration/Travel Time (RAW)
Travel rates by creature/vehicle/mount must use RAW speeds (e.g., horses vs humans).
Encumbrance, terrain, and fatigue can modify travel time.
### 4.4 Timers and Deadlines
Hard deadlines are supported: - “You have 24 hours to save the princess.” - DM must surface elapsed time and remaining time at appropriate beats. - Deadline progress is deterministic and logged.
### 4.5 Lighting Changes Over Time

### 3.5 Suppression Tracing

Each suppression includes a structured reason code

### 4.3 State Diff Format
Use domain‑specific structured diffs (entity‑level deltas)
JSON Patch and full snapshots are forbidden
### 4.4 Deterministic Serialization
Trace serialization must be stable
Identical inputs must produce byte‑for‑byte identical replays

# Alignment, Divine Influence & Moral Trajectory
Status: Design Intent (Non-Binding)
Scope: This document defines how alignment functions as a living narrative and evaluative system across campaigns, sessions, and character arcs. It clarifies how player behavior, divine expectations, and long-term consequences interact—without imposing deterministic control or removing player agency.
## 1. Design Goals
Make alignment matter without restricting choice
Alignment must shape the world’s response, not limit player actions.
Support long-term character arcs
Alignment should drift gradually in response to consistent patterns of behavior.
Restore narrative relevance of gods and religion
Deities must function as active evaluators of behavior, not abstract labels.
Provide structure for narrative intelligence
The language model is given clear evaluative context so it can respond with meaningful tone, tension, and consequence.
## 2. Alignment as an Emergent Property
Alignment is not a rule constraint or permission system.
Alignment is an emergent summary of accumulated actions and outcomes.
Key implications: - Characters may act out of alignment. - Single actions do not redefine alignment. - Persistent behavior trends do.
Declared alignment at character creation represents intent, not destiny.
## 3. Alignment Evidence Record
Each player character maintains an Alignment Evidence Record, containing: - Significant moral decisions - Harm caused, allowed, or prevented - Betrayals, loyalty, mercy, cruelty - Sacrifices and self-interest
Evidence is: - Descriptive, not judgmental - Cited to session events - Evaluated over time, not per incident
## 4. Alignment Evaluation Process
### 4.1 Gradual Evaluation
Alignment is evaluated periodically, not continuously.
Evaluation windows span multiple sessions.
### 4.2 Threshold-Based Drift
Alignment shifts only when evidence crosses meaningful thresholds.
Shifts are surfaced explicitly to players when they occur.
### 4.3 Transparency
When alignment changes, the system explains why, citing prior actions.
No silent or retroactive alignment changes are allowed.
## 5. Divine Alignment & Religious Consequences
### 5.1 Deities as Value Systems
Each deity embodies specific values and alignment constraints.
Deities evaluate behavior, not labels.
### 5.2 Favor and Disfavor
Divine favor reflects alignment congruence over time.
Drift may result in:
Diminished blessings
Altered divine communication
Withdrawal of granted powers (per rules)
These are outcomes, not punishments.
## 6. Class-Specific Implications
Clerics, paladins, warlocks, and other divine casters are especially sensitive to alignment drift.
Consequences are applied narratively first, mechanically second.
Mechanical effects must be justified with recorded evidence.
## 7. Narrative Use in DM Responses
Alignment informs: - Tone of descriptions - NPC expectations and reactions - Internal conflict cues - Divine omens, silence, or approval
Alignment does not: - Dictate dialogue choices - Override player intent - Remove agency
## 8. Player Freedom & Moral Uncertainty
Players may pursue redemption, corruption, or ambiguity freely.
The system records outcomes without enforcing arcs.
Moral uncertainty is preserved unless patterns clarify intent.
## 9. Adult / Unrestricted Themes
Alignment evaluation is content-agnostic.
Mature themes (violence, betrayal, faith crises) are recorded without sanitization.
Moral weight is contextual, not censored.
## 10. Non-Goals
This system does not: - Enforce real-world morality - Judge player intent - Prevent any action - Replace RAW mechanics
## 11. Application to Project Development
Moving forward, this document implies: - Alignment tracking schemas must be evidence-based - Divine systems must reference evaluated alignment, not declarations - Narrative generation should reference alignment trends when relevant - No alignment-driven logic belongs in low-level resolvers
## 12. Design Summary
Alignment is treated as a slow-moving moral trajectory shaped by actions and consequences. Gods respond to lived behavior, not stated intent. This framework gives narrative intelligence the structure needed to produce meaningful, consistent, and consequence-rich storytelling over long campaigns.

# Architecture Decisions
This document records architectural intent and design rationale for the project.
Its purpose is to preserve why certain choices were made, even if implementation happens later or evolves.
## Decision 001 — Obsidian as Player-Facing Evidence Viewer (Not Source of Truth)
### Status
Documented (Intent Only) — Not yet enforced as a required workflow component.
### Context
During gameplay, players may challenge AI rulings (“rules lawyering”). Trust and acceptance increase significantly when rulings are supported by verbatim Rules-As-Written (RAW) citations that players can inspect themselves.
An AI explanation alone is insufficient for high-trust adjudication. Players need a transparent, third-party–style reference that shows: - The exact rule text - The source book - The page number
### Problem
If the AI is the only authority presenting rules: - Players may distrust decisions as hallucinated or misinterpreted - Disputes become arguments with the AI rather than verification against text - Ambiguity in RAW feels arbitrary rather than explicit
Using Obsidian as a primary data store caused tight coupling, fragility, and uncertainty about what constituted the “true” rule source.
### Decision
Canonical rule data lives in a tool-agnostic source layer (sources/)
Obsidian is used as a generated, player-facing evidence viewer
Obsidian vaults are derived artifacts, not authoritative storage
The AI / engine may reference Obsidian notes only as a presentation layer for evidence, never as the origin of truth.
### Rationale
Obsidian excels at human-readable presentation, linking, and navigation
Generated vaults are disposable and regenerable
Showing players a readable, independent-looking reference increases trust
Separating storage from viewing prevents accidental corruption of canonical data
Disputes shift from “the AI decided” to “here is the printed rule”
### Intended Usage Pattern (Future)
Canonical text is stored in sources/ (meta, text, provenance)
An Obsidian vault is generated from that data
When the engine makes a ruling, it emits structured citations:
Book
Page
Text span

# D&D 3.5 Combat DM Decision Model: What the DM Evaluates and How Choices Are Made
## DECOMPOSE
To quantify DM decision-making in combat, model it as: 1) Inputs observed (state/features) 2) Objectives (what the DM is trying to optimize) 3) Constraints (rules + table goals) 4) Evaluation functions (scoring/utility per option) 5) Policy (how the DM selects actions: deterministic heuristics + stochastic spice) 6) Update loop (re-assess after each resolution event)
## SOLVE
## 1) DM decision inputs (the feature set)
Think of the DM as evaluating a feature vector each decision point.
### 1.1 Encounter-level features
Encounter purpose: tutorial, resource tax, boss, chase, puzzle-combat, narrative beat.
Win conditions: kill, delay, escape, protect, disrupt ritual, hold ground.
Expected length: target rounds.
Reinforcement schedule: timers/triggers.
Environment: terrain topology, hazards, visibility, movement channels.
### 1.2 Per-side macro features
Action economy: actions/round available; presence of summons/minions; control effects suppressing actions.
Threat distribution: where damage/control is coming from (who is carrying).
Resource posture: party spell slots/charges/HP observed; monster limited-use abilities.
Morale posture: are enemies disciplined, frightened, fanatical, cunning.
### 1.3 Per-PC features (what the DM infers)
For each PC i: - EHPᵢ (effective hit points): current HP + temp HP + mitigation (DR/ER) + expected healing access. - Defense profile: AC/touch/ff, saves, immunities, concealment. - Offense profile: DPR vs AC, save DC potency vs monster saves, control toolkit. - Mobility profile: speed, flight, teleport, freedom of movement. - Role signature: striker/controller/support/tank/scout. - Vulnerability flags: low Will, low Reflex, no ranged option, poor mobility.
### 1.4 Per-monster features
For each monster m: - Goal/behavior: protect leader, kill healer, delay, steal item, escape. - Burst potential: max spike damage and how telegraphed it is. - Control suite: stun/paralysis/grapple/trip/fear. - Survivability: AC/HP/DR/ER/SR and escape tools. - Position leverage: reach, AoOs, flight, burrow, ranged. - Cooldown/limited use: breath, rage, spells.
### 1.5 Local tactical features (per decision)
Distances and reachable squares.
Threatened squares and AoO risk.
LoS/LoE availability.
Cover/concealment vs key attackers.
Flank opportunities.
Choke points / zone control.
Status timers: remaining rounds on buffs/debuffs.
## 2) What the DM is balancing (objective functions)
A strong DM implicitly optimizes a weighted combination of goals.
### 2.1 Core objectives (typical weights)
Fair danger: credible loss possible without feeling like a gotcha.
Agency: player choices meaningfully change outcomes.
Pacing/tempo: no dead rounds; spotlight distribution.
Rules integrity: outcomes feel consistent with mechanics.
Fiction integrity: monsters behave intelligibly for their nature.
Represent as a utility function: - U = w₁·Danger + w₂·Agency + w₃·Pace + w₄·Rules + w₅·Fiction Weights shift by encounter purpose.
### 2.2 Danger (quantifiable proxies)
TPK risk (should be low except apex fights).
PC drop probability within next 1–2 rounds.
Resource burn: expected HP/spell loss per round.
Action denial risk: probability a PC loses turns.
### 2.3 Agency (quantifiable proxies)
Counterplay availability: number of viable responses to threats.
Information telegraphing: threat indicators preceding lethal spikes.
Outcome sensitivity: how much choices (positioning/targeting) change win odds.
### 2.4 Pace (quantifiable proxies)
Decision density: meaningful choices per round.
Resolution latency: time to resolve a full turn.
Swing moments frequency: exciting state changes every 1–2 rounds.
## 3) DM thought process per decision point (policy steps)
At each monster turn (and at key triggers), a competent DM runs roughly:
### Step 1: Reconstruct state
Update conditions, durations, positions.
Identify immediate threats and immediate vulnerabilities.
### Step 2: Identify “must address” constraints
If leader is exposed, protect/withdraw.
If objective is time-based, advance objective.
If monster morale breaks, flee/surrender.
### Step 3: Generate candidate actions
Candidates include: - Damage line: focus-fire a target. - Control line: disable a key PC. - Reposition line: gain cover, break LoE, set flanks. - Objective line: interact with object/escape/trigger hazard. - Defensive line: total defense, withdraw, heal, buff.
### Step 4: Score each candidate (quantification)
For each candidate action a, compute: - P_hit(a) or P_success(a) (attack vs AC; save DC vs save; opposed check). - E_damage(a) or E_effect(a) (expected value, including mitigation). - ΔThreat(a): change in enemy lethality next round. - ΔSurvival(a): change in monster survival probability. - ΔPosition(a): tactical leverage gain. - ΔAgency(a): does it create counterplay or remove it (too much lock)? - Rules risk(a): chance of adjudication complexity/slowdown. - Fiction fit(a): does the monster “do this”? (int score, instincts)
A simple utility score: - Score(a) = α·E_outcome + β·Leverage + γ·Survival − δ·Unfairness − ε·Complexity
### Step 5: Select with controlled randomness
Choose max score, but add variability:
Softmax over scores (prevents predictable “robot DM”).
Or “top-2” random choice when close.
### Step 6: Execute and narrate (telegraph)
Translate mechanics into fiction.
Telegraph lethal capabilities before they land.
## 4) Quantifying the math the DM is implicitly doing
### 4.1 Hit probability (weapon attacks)
Given attack bonus AB and target AC: - P_hit ≈ clamp((21 + AB − AC)/20, 0.05, 0.95) (Accounts for nat 1/20 bounds.)
### 4.2 Expected damage per attack
E_damage = P_hit · (AvgDamage_after_DR) · (1 + P_crit·(Mult−1)) · (1−MissChance) where:
P_crit depends on threat range and confirmation chance.
MissChance from concealment.
### 4.3 Save-based effect success
For DC and save bonus S: - P_fail ≈ clamp((DC − S − 1)/20, 0.05, 0.95) Then: - E_effect = P_fail·Full + (1−P_fail)·Partial
### 4.4 Action denial valuation
Denying a PC’s standard action is often worth ~1 PC’s expected DPR/control for a round. Define: - Value(deny turn of PC i) ≈ E_contributionᵢ_next_round This becomes a primary reason DMs use grapples, stuns, fear, etc.—but must be tempered by fairness/agency weights.
### 4.5 Focus fire vs spread damage
Focus fire maximizes probability of a drop (reduces enemy actions).
Spread damage maximizes “pressure” but usually preserves enemy actions. A DM chooses based on:
Desired danger and fiction (predators focus; undisciplined mobs spread).
## 5) How an expert DM keeps combat accurate + enjoyable + dangerous
### 5.1 Accuracy safeguards (procedural)
Use the fixed resolution order: legality → roll → gates (SR/miss) → apply → update.
Maintain a strict conditions/duration tick rule.
Precompute common probability bands (hit chance vs frontliner vs backliner).
### 5.2 Enjoyment safeguards (agency & pacing)
Prefer threats with counterplay over hard locks.
Use terrain/objectives to create choices beyond “attack again.”
Rotate spotlight: different PCs become relevant via movement lanes, saves, skills.
### 5.3 Danger safeguards (credible failure)
Enemies occasionally use:
focus fire,
action denial,
smart positioning,
retreat/regroup,
reinforcements.
But avoid “un-signaled” lethal spikes unless the table wants it.
### 5.4 “Fairness telemetry” (when to ease off)
Trigger conditions for easing difficulty: - Two PCs down early with no counterplay. - Players demonstrate confusion rather than poor choices. - Extreme variance (crits) creates runaway. Mechanisms: - Monster morale breaks. - Target switching consistent with fiction. - Tactical suboptimality that still makes sense (overconfidence).
## 6) Turning this into a quantifiable AI/engine (deliverable spec)
### 6.1 Define a DM Policy API
At each decision point provide: - state: full combat state - intent: encounter and monster goals - options: legal actions enumerated Return: - chosen_action - score_breakdown (danger/agency/pace/rules/fiction) - telegraph_text (narration hints)
### 6.2 Minimal feature set for a first working model
P_hit, E_damage, P_fail for save DCs
Action economy delta
Position leverage (cover, flanks, AoO risk)
Counterplay score
Complexity score
### 6.3 Calibration knobs (what you tune)
Encounter weight vector w
Monster personality vector (aggression, caution, cruelty, intelligence)
Randomness temperature (predictability vs variety)
## VERIFY
This model maps DM “judgment” to a constrained optimization over probability of success, expected effects, and table-centric utility (fun/fairness/pacing).
The quant pieces (hit chance, save chance, expected value) are simplified but correct enough to drive a policy layer; a full engine would compute exact probabilities with d20 discrete distributions.
## SYNTHESIZE
An expert 3.5 DM is continuously balancing: - mechanical advantage (expected value and action economy) - survivability (not suiciding monsters unless fiction demands) - player experience (agency, pacing, fairness) - fictional coherence (monster intelligence/morale)
This can be quantified as a scoring function over candidate actions, updated each turn, with calibrated weights and controlled randomness.
# Secondary Accuracy + Completeness Audit (DM Decision Quantification)
## A) Precision corrections (math + terminology)
### A1) Probability clamp bounds (attack rolls)
Using 0.05–0.95 is appropriate for attack rolls because nat 1/nat 20 bound hit/miss probability.
But attack-roll hit probability is not exactly linear when you include:
differing AC types (touch/ff),
auto-hit on 20 even if AC is unreachable,
and auto-miss on 1 even if AB >> AC.
The approximation is acceptable for policy, but label it as approx.
Confidence: 0.9
### A2) Save probabilities must not use nat 1/20 bounds (RAW)
In 3.5 RAW, saves do not auto-fail on 1 or auto-succeed on 20.
Therefore do not clamp save success/fail probabilities to 0.05–0.95 unless you explicitly adopt a house rule.
A better discrete approximation:
P_fail = count{r in 1..20 : r + S < DC} / 20 (no special cases)
Confidence: 0.95
### A3) Crit expected value needs explicit confirmation chance
P_crit must be modeled as:
P_threat (threat range) × P_confirm (second roll hits AC)
If concealment exists, it can cancel a confirmed crit because the attack misses.
Confidence: 0.9
### A4) Miss chance gating order (implementation contract)
For a computable engine, you must specify whether concealment is applied:
before confirmation,
after confirmation,
or on each attack instance.
Table practice varies; choose one contract and keep it consistent.
Confidence: 0.75
### A5) Action denial valuation is role-dependent
“Deny a standard action ≈ 1 round of DPR/control” is a good heuristic.
But it must be conditioned on:
PC role (controller vs striker),
current resources (nova vs depleted),
and board state (is the PC in position to contribute?).
So define E_contributionᵢ_next_round as a computed feature, not a constant.
Confidence: 0.85
## B) Missing or under-modeled decision elements (added)
### B1) Target identification uncertainty (major omission)
A DM decision policy must model beliefs, not only ground-truth: - Hidden defenses: unknown saves, DR/ER, SR, immunities. - Unknown HP (by design). - Unknown spells remaining. Quantification: - Maintain per-PC and per-monster belief distributions over key parameters (e.g., “Will save likely low”). - Update beliefs via observed outcomes (Bayesian-ish).
Confidence: 0.8
### B2) Risk management vs variance control
DMs manage not just expected value, but variance: - Avoiding swingy lines (rocket-tag) except when desired. Add a term: - − κ·Var(outcome) (or control the “temperature” by encounter type).
Confidence: 0.8
### B3) Multi-turn planning horizon (lookahead)
A DM often optimizes over 2–3 rounds: - set up flanks, - bait AoOs, - force movement into hazards, - wait for breath recharge. Add: - Score(a) = immediate + λ·expected_future_value (small depth lookahead).
Confidence: 0.75
### B4) Initiative position manipulation (ready/delay) as a decision axis
Ready/delay changes not only who acts when but can create “double turns” or deny healing windows. Add features:
ΔInitiativeAdvantage and window_denial.
Confidence: 0.85
### B5) Concentration + disruption as explicit evaluation
Against casters, DM decisions include: - provoke AoOs, - force concentration checks, - apply silence/grapple, - line-of-effect denial. This is distinct from generic “control.”
Confidence: 0.85
### B6) Social contract constraints (table-level)
A DM is constrained by: - spotlight fairness, - not invalidating builds repeatedly, - acceptable lethality, - session time budget. These must be explicit weights/guardrails (hard constraints or penalties).
Confidence: 0.8
### B7) Monster doctrine/personality as a policy prior
A monster’s intelligence/morale creates a prior policy: - mindless: local greedy, - disciplined: focus fire, - cowardly: withdraw when bloodied, - fanatic: fight to death. Quantification: - personality vector p transforms weights and candidate generation.
Confidence: 0.85
### B8) Surrender/flee/negotiation decision branch
Combat decisions include “end combat” choices: - flee, yield, bargain, take hostage. Quantification: - add “exit actions” to candidate set when morale thresholds reached.
Confidence: 0.8
### B9) Party feedback loop (player psychology)
DMs adjust intensity based on observed: - confusion vs mastery, - boredom, - stress, - time remaining. Quantify as observed signals: - engagement, clarity, time_budget and use as pacing modifiers.
Confidence: 0.7
## C) Completeness checklist for a DM decision engine (updated)
To be “nothing left out” for decision-making (not rule resolution), you need: 1) State features (positions, conditions, resources) 2) Legal action enumerator (RAW constraints) 3) Candidate generator (damage/control/objective/defense/exit) 4) Probability models (attacks, saves, opposed, concealment, SR) 5) Belief model (unknown enemy/PC parameters + update) 6) Utility function (danger/agency/pace/rules/fiction) 7) Variance/risk control term 8) Planning horizon / lookahead 9) Personality prior (monster doctrine) 10) Social-contract guardrails (spotlight/time/lethality limits) 11) Selection policy (argmax + softmax randomness) 12) Narration/telegraph layer (information design)

# D&D 3.5e — Deterministic Mechanics Audit (Program-Implementable)
## Scope
Rules-as-written, deterministic / procedurally defined subsystems that can be implemented as code. - Deterministic here includes: explicit formulas, tables, DCs, step-by-step procedures, and RNG inputs (dice). - Excludes: pure narration, GM judgment that is explicitly undefined (e.g., “the DM decides the NPC’s goals”), and optional house rules.
# 1) Global Math & Stacking Rules
### 1.1 Dice / RNG primitives
Dice notation: NdS + K (sum of N uniform integers in [1..S] plus constant K).
d% / percentile = uniform integer in [1..100].
Advantage/disadvantage does not exist in 3.5 core.
### 1.2 Rounding & fractions
Round down on fractional results unless explicitly stated otherwise.
Conversions that halve/quarter values follow floor behavior.
### 1.3 Bonus types & stacking
Typed bonuses: only highest of a given type applies; different types stack.
Untyped bonuses generally stack unless from same source.
Penalties stack unless specified.
Exceptions to “no same-type stacking”: dodge bonuses stack; circumstance stack if from different circumstances.
### 1.4 Multipliers
Multiple multipliers combine additively (×2 then ×2 → ×3), unless rules specify otherwise.
### 1.5 Contested vs. fixed target numbers
Resolution pattern categories:
Check vs DC: d20 + mod ≥ DC.
Opposed checks: d20 + mod vs d20 + mod.
Attack vs AC: d20 + attack bonus ≥ AC.
Caster level check vs SR: d20 + CL ≥ SR.
Secondary accuracy check: These are core SRD principles and are consistent.
# 2) Character Generation & Progression Systems
## 2.1 Ability scores
Generation methods (roll, point buy, array) → score values.
Ability modifier: floor((score − 10)/2).

## 2.2 Race
Size, speed, ability adjustments, skill bonuses, weapon familiarity, special abilities.
## 2.3 Class
Hit die, BAB progression, save progressions, skill points/level, class skills, proficiencies, spells/day, class features.

## 2.5 Feats

XP penalties (core rule variant), favored class interactions.
Secondary evaluation (omissions): Include alignment restrictions, code of conduct (paladin/monk), and class feature loss/recovery procedures.
# 3) Combat System (Turn Structure + Actions)

Rounds (6 seconds), initiative order, surprise round.
## 3.2 Initiative

# D&D 3.5 Combat: Elements, Tracking, and Mathematics (Start → Finish)
## DECOMPOSE
A full combat analysis breaks into: 1) State to track (DM vs players) 2) Round/turn algorithm (initiative → actions → resolution) 3) Math checks (attack/defense, saves, skills, concealment, grapples, etc.) 4) Resource & effect accounting (HP, conditions, durations, actions, positioning) 5) What makes it exciting (stakes, tempo, uncertainty, tactical choices) 6) How to run it accurately + enjoyably (procedural discipline + pacing tools)
## SOLVE
## 1) Combat state: everything that exists “on the board”
### 1.1 Entities
Combatants: PCs, NPCs, monsters, summons, cohorts, companions.
Objects: doors, cover objects, traps, hazards, vehicles, siege engines.
Zones: difficult terrain, elevation, water, darkness, magical areas.
### 1.2 Spatial model
Track per entity: - Grid position (square/hex) or relative distances. - Facing (optional; default 3.5 doesn’t require, but some tables use it). - Space (size category footprint). - Reach (threatened squares). - Movement mode(s): land, fly (with maneuverability), swim, burrow, climb. - Terrain interaction: difficult terrain costs; obstacles; squeezing; elevation. - Line of sight (LoS) and line of effect (LoE). - Cover category and concealment category vs each attacker.
### 1.3 Time model
Initiative order.
Current round count.
Durations: effects measured in rounds/minutes/hours, with start/end triggers.
### 1.4 Information model
Public info (seen by players): positions (usually), visible conditions.
Private info (DM-only): hidden enemies, traps, exact HP, secret DCs.
Uncertainty sources: miss chance, concealment, random damage, morale.
## 2) What the DM tracks (combat ledger)
### 2.1 Per creature (DM-facing)
Identity & rules block - Creature name, type/subtype, size. - Speed(s), senses (darkvision, blindsense, scent). - AC breakdown: 10 + armor + shield + Dex + size + natural + deflection + misc. - Touch AC, Flat-footed AC. - HP (current/max), temp HP. - DR, energy resistance, immunities. - Saving throws: Fort/Ref/Will totals (and conditional modifiers). - Attack routines: melee/ranged bonuses, iterative attacks, natural attacks. - Damage expressions incl. bonuses, crit range/multiplier. - Special attacks: grapple, trip, poison, breath, etc. - Spell-like/spells: DCs, remaining slots/uses, concentration.
Dynamic combat state - Initiative value. - Action economy state this turn: standard/move/swift/immediate, free, AoOs used. - Current weapon/stance, whether drawing occurred, whether 5-ft step used. - Conditions list (stunned, prone, grappled, etc.). - Ongoing effects: buffs/debuffs with remaining duration. - Status toggles: concealed? invisible? mirror image count? stoneskin pool? rage rounds? - Positioning flags: threatened squares, flanking relationships. - Morale/intelligence behavior triggers.
### 2.2 Battlefield + encounter-level
Map features + their rules effects (cover lines, difficult terrain, hazards).
Environmental rules: lighting, wind, underwater, fog.
Encounter triggers: reinforcements, alarms, collapsing floors, time pressure.
Victory/defeat conditions (not always “kill all”).
### 2.3 Hidden bookkeeping
Monster tactics script (if running fast): “If X then Y”.
XP/treasure (optional during combat).
Secret perception/listen/spot DCs.
## 3) What the player tracks
### 3.1 Character sheet values used constantly
Attack bonuses (by weapon; full attack sequence).
AC (normal/touch/flat-footed) and situational modifiers.
HP and temp HP.
Saves and condition modifiers.
Speed and movement options.
Feats that change math (Power Attack, Combat Expertise, Rapid Shot, etc.).
Spell DCs, spell slots, prepared spells, concentration.
Resources: rage uses, bardic music, smite, turn undead, ki-like pools (if any).
### 3.2 Turn-by-turn tactical state
Where they are (position).
What they can threaten (reach).
Whether they have used: 5-ft step, AoO(s), swift action.
Active effects and remaining durations.
Ammo/charges, consumables.
### 3.3 Information the player infers
Enemy apparent AC “band,” resistances, save weaknesses (via feedback).
Map affordances (cover lanes, choke points).
Risk estimates (chance to hit, chance to drop, etc.).
## 4) The combat algorithm (start → finish)
### 4.1 Combat start
Establish positions and surprise conditions.
Surprise round: only surprised? only non-surprised act; standard or move action only.
Roll initiative for all: d20 + initiative modifier.
Build initiative order; note tie-breaks.
### 4.2 Each round
Repeat until encounter ends: 1) Round counter increments. 2) Each combatant acts in initiative order.
### 4.3 Each turn (skeleton)
For the active creature: 1) Apply start-of-turn effects (ongoing damage, regen, etc.). 2) Choose action package: - Standard + move (typical) - Full-round (full attack, charge, withdraw, run) - plus swift/immediate, free, and AoOs outside turn. 3) Resolve movement (distance, terrain costs, AoO triggers). 4) Resolve attacks/spells/special actions (math below). 5) Apply end-of-turn effects and decrement durations.
## 5) The mathematics: every major check category
### 5.1 Dice primitives
d20 tests: d20 + modifiers vs target number.
Damage dice: sum of dice + static bonuses.
Percentile / d100: sometimes (e.g., spell failure tables, random direction).
### 5.2 Attack resolution (weapon/natural)
For each attack roll: - Compute Attack Bonus: - BAB + ability mod (Str melee, Dex ranged; finesse exceptions) + size + enhancement + misc. - Apply penalties: Power Attack? Combat Expertise? Rapid Shot? Two-Weapon Fighting? range increment? cover? concealment? prone? grappling? - Roll d20; apply auto miss/hit (1/20). - Compare vs target AC (normal). If touch attack, compare vs touch AC. - If hit: - Roll damage: weapon dice + Str (or 1.5x for two-handed) + enhancement + Power Attack bonus + misc. - Apply multipliers (crit) and add extra dice that are not multiplied (as applicable). - Apply mitigation: DR, energy resistance, immunities.
Critical hits - Threat range: e.g., 19–20. - Confirm: second attack roll vs AC. - On confirm: multiply damage by crit multiplier (typically x2/x3/x4), excluding many bonus dice.
### 5.3 Armor Class & being harder to hit
AC = 10 + armor + shield + Dex + size + natural + deflection + misc.
Touch AC strips armor/shield/natural (mostly), keeps Dex/deflection.
Flat-footed removes Dex bonuses.
Situational modifiers: cover (+4), concealment (miss chance), prone (+4 vs ranged, -4 vs melee), helpless (Dex 0; melee hits often auto-crit rules apply).
### 5.4 Saving throws
For an effect requiring a save: - Roll d20 + save bonus (Fort/Ref/Will) + situational. - Compare vs DC. - DC typically: 10 + spell level + casting ability mod + feat/item mods. - Apply outcomes: negate, half, partial, etc.
### 5.5 Skill checks used in combat
Common: - Concentration: d20 + ranks + Con + misc vs DC (damage taken, vigorous motion, etc.). - Tumble: avoid AoOs, move through threatened squares. - Hide/Move Silently vs Spot/Listen. - Balance/Climb/Swim under hazards. - Use Magic Device during item use.
### 5.6 Initiative & action economy
Initiative: d20 + Dex mod + misc.
Action constraints:
Standard: attack, cast, etc.
Move: move up to speed, stand from prone, draw weapon (often), etc.
Full-round: full attack, charge, withdraw.
Swift/Immediate: limited special actions.
Rules interactions: provoking AoOs; casting defensively; readied actions; delays.
### 5.7 Attacks of Opportunity (AoO)
Triggered by: leaving threatened square, casting, ranged attacks in melee, standing up, etc.
Limit: 1 per round normally (more with Combat Reflexes + Dex).
Resolution: normal attack roll vs AC.
### 5.8 Grapple subsystem (notorious)
Key values: - Grapple modifier = BAB + Str mod + size mod + misc. Workflow: 1) Make melee touch attack to grab. 2) Opposed grapple check to establish hold. 3) Each round: opposed grapple checks to damage, pin, move, escape, etc. Also track: losing Dex to AC, limited actions, spellcasting restrictions.
### 5.9 Trip, disarm, sunder, bull rush, overrun
Each has a specific opposed-check math pattern, generally: - Touch attack (sometimes) → opposed ability check (Str vs Str or Dex/Str) + size + misc. - Apply positional outcomes: prone (trip), weapon dropped (disarm), object damage (sunder), forced movement (bull rush).
### 5.10 Concealment, invisibility, miss chances
Concealment: attacker hits AC first, then roll miss chance (20%/50%).
True seeing/blindsense can nullify.
### 5.11 Spellcasting resolution
For spells in combat: - Determine target legality via LoS/LoE, range, area. - Handle SR (Spell Resistance) check: d20 + caster level vs SR. - Handle save, if any. - Apply duration tracking (rounds/level, minutes/level). - Handle concentration if threatened or damaged.
### 5.12 Damage, death, stabilization
Track HP dropping below 0:
-1 to -9: dying; lose 1 HP/round unless stabilized.
-10: dead.
Stabilization checks, healing timing.
Massive damage (if used).
### 5.13 Ongoing damage, regen, fast healing
Apply at start/end as specified.
Track special bypass conditions for regen.
### 5.14 Conditions (each changes math)
Minimum list to treat as “math-changing”: - Blinded, dazzled - Deafened - Entangled - Exhausted/fatigued - Frightened/panicked/shaken - Grappled/pinned - Helpless - Invisible - Nauseated/sickened - Paralyzed - Prone - Stunned - Turned - Confused - Slowed/haste Each condition modifies: attack rolls, AC, actions, speed, saves, miss chance.
## 6) DM accuracy: the “combat accounting system”
### 6.1 Minimal ledger (fast + correct)
For every creature keep a single line: - Init | HP | AC (and touch/ff) | Saves | Attack bonus & dmg | Speed | Conditions | Durations | AoO remaining
### 6.2 Deterministic procedure order (reduces errors)
When resolving any effect, always do: 1) Legality (range, LoS/LoE, target type) 2) Attack roll OR save OR opposed check 3) Secondary gates (SR, concealment miss, crit confirm) 4) Damage/effect application (including mitigation) 5) Update state (HP, conditions, durations, position)
### 6.3 “Math compression” techniques
Precompute common totals: melee +X, ranged +Y, primary save DCs.
Use condition tokens/cards; decrement durations at consistent phase.
Group identical monsters and use average damage or batch rolls if table accepts.
Track monster HP as “bands” internally if you want narrative uncertainty while staying rules-legal (see §7.4).
## 7) What makes combat exciting (and still dangerous)
### 7.1 Excitement is decision density + stakes + tempo
Key drivers: - Meaningful choices: multiple viable lines (positioning, target priority, resource timing). - Visible stakes: something is lost if players stall (hostage, ritual, fleeing villain). - Tempo variance: alternating bursts (big hits) and control (movement, terrain shifts). - Information drip: players learn enemy capabilities via play.
### 7.2 Danger comes from credible failure modes
Danger is real when: - Players can be outflanked, separated, status-locked, resource-drained. - Healing has opportunity cost. - Enemies use focus fire and action denial sometimes.
### 7.3 Fairness comes from telegraphing and counterplay
Telegraph lethal threats 1–2 beats early.
Ensure counterplay exists: cover, retreat lanes, dispel options, saving throw supports.
### 7.4 Keeping HP hidden without “video-game bars”
Rules-legal presentation: - DM tracks exact HP. - Players receive descriptive states (fresh, scratched, bloodied, staggering). - Optionally map to thresholds (e.g., >75%, 75–50, 50–25, <25) for consistent narration.
### 7.5 Encounter design levers that are mathematical
Action economy: number of meaningful actions per side per round.
Accuracy bands: expected chance-to-hit (~35–65% feels interactive).
Save DC vs save bonuses: avoid “no-save” lockouts unless telegraphed.
Damage pacing: average DPR vs party HP pool; spikes via crits/smites.
Terrain: creates asymmetric advantages and tactical puzzles.
Waves/reinforcements: changes action economy mid-fight.
## 8) End condition and post-combat math
End triggers: surrender, flee, objective met, TPK, escape.
Stabilization/healing cleanup.
Resource accounting: consumables used, spell slots, charges.
Loot/time passage/XP (often after).

# Secondary Accuracy + Completeness Audit (3.5 Combat)
## A) Corrections / Precision fixes
### A1) Natural armor and touch AC (precision)
Touch AC removes armor, shield, and natural armor bonuses (and most other “physical” bonuses), but keeps Dex, deflection, and some dodge/misc bonuses.
In the main draft I wrote “touch strips armor/shield/natural (mostly)” — this is directionally right, but the precise statement is: natural armor is not counted in touch AC.
### A2) Auto 1/20 scope (precision)
Natural 1 / natural 20 auto-miss/auto-hit applies to attack rolls only.
It does not automatically fail/succeed on saving throws, skill checks, or ability checks (unless a table house-rules otherwise).
### A3) Coup de grace and crit language (precision)
“Helpless → melee hits often auto-crit” is not accurate phrasing.
The correct RAW mechanism is coup de grace: a full-round action that is an automatic critical hit against a helpless target, followed by a Fort save vs DC 10 + damage dealt or die.
### A4) Damage multiplication nuance (precision)
Many bonus dice (e.g., sneak attack) are not multiplied on a crit.
Static bonuses usually multiply if they are part of weapon damage, but always follow the specific effect’s rule text.
### A5) Duration decrement convention (table discipline)
RAW effects specify their timing; a “decrement at end of turn” rule is a good implementation convention, but should be documented as a house bookkeeping standard to avoid drift.

Morale posture: are enemies disciplined, frightened, fanatical, cunning.
1.3 Per-PC features (what the DM infers)
For each PC i: - EHPᵢ (effective hit points): current HP + temp HP + mitigation (DR/ER) + expected healing access. - Defense profile: AC/touch/ff, saves, immunities, concealment. - Offense profile: DPR vs AC, save DC potency vs monster saves, control toolkit. - Mobility profile: speed, flight, teleport, freedom of movement. - Role signature: striker/controller/support/tank/scout. - Vulnerability flags: low Will, low Reflex, no ranged option, poor mobility.
1.4 Per-monster features
For each monster m: - Goal/behavior: protect leader, kill healer, delay, steal item, escape. - Burst potential: max spike damage and how telegraphed it is. - Control suite: stun/paralysis/grapple/trip/fear. - Survivability: AC/HP/DR/ER/SR and escape tools. - Position leverage: reach, AoOs, flight, burrow, ranged. - Cooldown/limited use: breath, rage, spells.
1.5 Local tactical features (per decision)
Distances and reachable squares.
Threatened squares and AoO risk.
LoS/LoE availability.
Cover/concealment vs key attackers.
Flank opportunities.
Choke points / zone control.
Status timers: remaining rounds on buffs/debuffs.
2) What the DM is balancing (objective functions)
A strong DM implicitly optimizes a weighted combination of goals.
2.1 Core objectives (typical weights)
Fair danger: credible loss possible without feeling like a gotcha.
Agency: player choices meaningfully change outcomes.
Pacing/tempo: no dead rounds; spotlight distribution.
Rules integrity: outcomes feel consistent with mechanics.
Fiction integrity: monsters behave intelligibly for their nature.
Represent as a utility function: - U = w₁·Danger + w₂·Agency + w₃·Pace + w₄·Rules + w₅·Fiction Weights shift by encounter purpose.
2.2 Danger (quantifiable proxies)
TPK risk (should be low except apex fights).
PC drop probability within next 1–2 rounds.
Resource burn: expected HP/spell loss per round.
Action denial risk: probability a PC loses turns.
2.3 Agency (quantifiable proxies)
Counterplay availability: number of viable responses to threats.
Information telegraphing: threat indicators preceding lethal spikes.
Outcome sensitivity: how much choices (positioning/targeting) change win odds.
2.4 Pace (quantifiable proxies)
Decision density: meaningful choices per round.
Resolution latency: time to resolve a full turn.
Swing moments frequency: exciting state changes every 1–2 rounds.
3) DM thought process per decision point (policy steps)
At each monster turn (and at key triggers), a competent DM runs roughly:
Step 1: Reconstruct state
Update conditions, durations, positions.
Identify immediate threats and immediate vulnerabilities.
Step 2: Identify “must address” constraints
If leader is exposed, protect/withdraw.
If objective is time-based, advance objective.
If monster morale breaks, flee/surrender.
Step 3: Generate candidate actions
Candidates include: - Damage line: focus-fire a target. - Control line: disable a key PC. - Reposition line: gain cover, break LoE, set flanks. - Objective line: interact with object/escape/trigger hazard. - Defensive line: total defense, withdraw, heal, buff.
Step 4: Score each candidate (quantification)
For each candidate action a, compute: - P_hit(a) or P_success(a) (attack vs AC; save DC vs save; opposed check). - E_damage(a) or E_effect(a) (expected value, including mitigation). - ΔThreat(a): change in enemy lethality next round. - ΔSurvival(a): change in monster survival probability. - ΔPosition(a): tactical leverage gain. - ΔAgency(a): does it create counterplay or remove it (too much lock)? - Rules risk(a): chance of adjudication complexity/slowdown. - Fiction fit(a): does the monster “do this”? (int score, instincts)
A simple utility score: - Score(a) = α·E_outcome + β·Leverage + γ·Survival − δ·Unfairness − ε·Complexity
Step 5: Select with controlled randomness
Choose max score, but add variability:
Softmax over scores (prevents predictable “robot DM”).
Or “top-2” random choice when close.
Step 6: Execute and narrate (telegraph)
Translate mechanics into fiction.
Telegraph lethal capabilities before they land.
4) Quantifying the math the DM is implicitly doing
4.1 Hit probability (weapon attacks)
Given attack bonus AB and target AC: - P_hit ≈ clamp((21 + AB − AC)/20, 0.05, 0.95) (Accounts for nat 1/20 bounds.)
4.2 Expected damage per attack
E_damage = P_hit · (AvgDamage_after_DR) · (1 + P_crit·(Mult−1)) · (1−MissChance) where:
P_crit depends on threat range and confirmation chance.
MissChance from concealment.
4.3 Save-based effect success
For DC and save bonus S: - P_fail ≈ clamp((DC − S − 1)/20, 0.05, 0.95) Then: - E_effect = P_fail·Full + (1−P_fail)·Partial
4.4 Action denial valuation
Denying a PC’s standard action is often worth ~1 PC’s expected DPR/control for a round. Define: - Value(deny turn of PC i) ≈ E_contributionᵢ_next_round This becomes a primary reason DMs use grapples, stuns, fear, etc.—but must be tempered by fairness/agency weights.
4.5 Focus fire vs spread damage
Focus fire maximizes probability of a drop (reduces enemy actions).
Spread damage maximizes “pressure” but usually preserves enemy actions. A DM chooses based on:
Desired danger and fiction (predators focus; undisciplined mobs spread).
5) How an expert DM keeps combat accurate + enjoyable + dangerous
5.1 Accuracy safeguards (procedural)
Use the fixed resolution order: legality → roll → gates (SR/miss) → apply → update.
Maintain a strict conditions/duration tick rule.
Precompute common probability bands (hit chance vs frontliner vs backliner).
5.2 Enjoyment safeguards (agency & pacing)
Prefer threats with counterplay over hard locks.
Use terrain/objectives to create choices beyond “attack again.”
Rotate spotlight: different PCs become relevant via movement lanes, saves, skills.
5.3 Danger safeguards (credible failure)
Enemies occasionally use:
focus fire,
action denial,
smart positioning,
retreat/regroup,
reinforcements.
But avoid “un-signaled” lethal spikes unless the table wants it.
5.4 “Fairness telemetry” (when to ease off)
Trigger conditions for easing difficulty: - Two PCs down early with no counterplay. - Players demonstrate confusion rather than poor choices. - Extreme variance (crits) creates runaway. Mechanisms: - Monster morale breaks. - Target switching consistent with fiction. - Tactical suboptimality that still makes sense (overconfidence).
6) Turning this into a quantifiable AI/engine (deliverable spec)
6.1 Define a DM Policy API
At each decision point provide: - state: full combat state - intent: encounter and monster goals - options: legal actions enumerated Return: - chosen_action - score_breakdown (danger/agency/pace/rules/fiction) - telegraph_text (narration hints)
6.2 Minimal feature set for a first working model
P_hit, E_damage, P_fail for save DCs
Action economy delta
Position leverage (cover, flanks, AoO risk)
Counterplay score
Complexity score
6.3 Calibration knobs (what you tune)
Encounter weight vector w
Monster personality vector (aggression, caution, cruelty, intelligence)
Randomness temperature (predictability vs variety)
VERIFY
This model maps DM “judgment” to a constrained optimization over probability of success, expected effects, and table-centric utility (fun/fairness/pacing).
The quant pieces (hit chance, save chance, expected value) are simplified but correct enough to drive a policy layer; a full engine would compute exact probabilities with d20 discrete distributions.
SYNTHESIZE
An expert 3.5 DM is continuously balancing: - mechanical advantage (expected value and action economy) - survivability (not suiciding monsters unless fiction demands) - player experience (agency, pacing, fairness) - fictional coherence (monster intelligence/morale)
This can be quantified as a scoring function over candidate actions, updated each turn, with calibrated weights and controlled randomness.

A) Precision corrections (math + terminology)
A1) Probability clamp bounds (attack rolls)
Using 0.05–0.95 is appropriate for attack rolls because nat 1/nat 20 bound hit/miss probability.
But attack-roll hit probability is not exactly linear when you include:
differing AC types (touch/ff),
auto-hit on 20 even if AC is unreachable,
and auto-miss on 1 even if AB >> AC.
The approximation is acceptable for policy, but label it as approx.
Confidence: 0.9
A2) Save probabilities must not use nat 1/20 bounds (RAW)
In 3.5 RAW, saves do not auto-fail on 1 or auto-succeed on 20.
Therefore do not clamp save success/fail probabilities to 0.05–0.95 unless you explicitly adopt a house rule.
A better discrete approximation:
P_fail = count{r in 1..20 : r + S < DC} / 20 (no special cases)
Confidence: 0.95
A3) Crit expected value needs explicit confirmation chance
P_crit must be modeled as:
P_threat (threat range) × P_confirm (second roll hits AC)
If concealment exists, it can cancel a confirmed crit because the attack misses.
Confidence: 0.9
A4) Miss chance gating order (implementation contract)
For a computable engine, you must specify whether concealment is applied:
before confirmation,
after confirmation,
or on each attack instance.
Table practice varies; choose one contract and keep it consistent.
Confidence: 0.75
A5) Action denial valuation is role-dependent
“Deny a standard action ≈ 1 round of DPR/control” is a good heuristic.
But it must be conditioned on:
PC role (controller vs striker),
current resources (nova vs depleted),
and board state (is the PC in position to contribute?).
So define E_contributionᵢ_next_round as a computed feature, not a constant.
Confidence: 0.85
B) Missing or under-modeled decision elements (added)
B1) Target identification uncertainty (major omission)
A DM decision policy must model beliefs, not only ground-truth: - Hidden defenses: unknown saves, DR/ER, SR, immunities. - Unknown HP (by design). - Unknown spells remaining. Quantification: - Maintain per-PC and per-monster belief distributions over key parameters (e.g., “Will save likely low”). - Update beliefs via observed outcomes (Bayesian-ish).
Confidence: 0.8
B2) Risk management vs variance control
DMs manage not just expected value, but variance: - Avoiding swingy lines (rocket-tag) except when desired. Add a term: - − κ·Var(outcome) (or control the “temperature” by encounter type).
Confidence: 0.8
B3) Multi-turn planning horizon (lookahead)
A DM often optimizes over 2–3 rounds: - set up flanks, - bait AoOs, - force movement into hazards, - wait for breath recharge. Add: - Score(a) = immediate + λ·expected_future_value (small depth lookahead).
Confidence: 0.75
B4) Initiative position manipulation (ready/delay) as a decision axis
Ready/delay changes not only who acts when but can create “double turns” or deny healing windows. Add features:
ΔInitiativeAdvantage and window_denial.
Confidence: 0.85
B5) Concentration + disruption as explicit evaluation
Against casters, DM decisions include: - provoke AoOs, - force concentration checks, - apply silence/grapple, - line-of-effect denial. This is distinct from generic “control.”
Confidence: 0.85
B6) Social contract constraints (table-level)
A DM is constrained by: - spotlight fairness, - not invalidating builds repeatedly, - acceptable lethality, - session time budget. These must be explicit weights/guardrails (hard constraints or penalties).
Confidence: 0.8
B7) Monster doctrine/personality as a policy prior
A monster’s intelligence/morale creates a prior policy: - mindless: local greedy, - disciplined: focus fire, - cowardly: withdraw when bloodied, - fanatic: fight to death. Quantification: - personality vector p transforms weights and candidate generation.
Confidence: 0.85
B8) Surrender/flee/negotiation decision branch
Combat decisions include “end combat” choices: - flee, yield, bargain, take hostage. Quantification: - add “exit actions” to candidate set when morale thresholds reached.
Confidence: 0.8
B9) Party feedback loop (player psychology)
DMs adjust intensity based on observed: - confusion vs mastery, - boredom, - stress, - time remaining. Quantify as observed signals: - engagement, clarity, time_budget and use as pacing modifiers.
Confidence: 0.7
C) Completeness checklist for a DM decision engine (updated)
To be “nothing left out” for decision-making (not rule resolution), you need: 1) State features (positions, conditions, resources) 2) Legal action enumerator (RAW constraints) 3) Candidate generator (damage/control/objective/defense/exit) 4) Probability models (attacks, saves, opposed, concealment, SR) 5) Belief model (unknown enemy/PC parameters + update) 6) Utility function (danger/agency/pace/rules/fiction) 7) Variance/risk control term 8) Planning horizon / lookahead 9) Personality prior (monster doctrine) 10) Social-contract guardrails (spotlight/time/lethality limits) 11) Selection policy (argmax + softmax randomness) 12) Narration/telegraph layer (information design)
Confidence: 0.83
D) Corrected formulas (engine-ready)
D1) Attack hit probability (approx)
P_hit ≈ clamp((21 + AB − AC)/20, 0.05, 0.95) (attack rolls only)
D2) Save fail probability (RAW discrete)
P_fail = |{r∈1..20 : r + S < DC}| / 20
D3) Crit probability
P_crit = P_threat × P_confirm (and apply miss-chance contract consistently)
D4) Expected value of action
EV(a) = P_success(a)·Value_full + (1−P_success)·Value_partial − cost

# Grapple System Amendment — RAW Fidelity & Phased Implementation
Status: Binding once merged
Applies To: CP-14 and all subsequent instruction packets
## 1. Purpose
This amendment formalizes the project’s handling of the grapple subsystem (PHB 3.5e, pp. 155–157) to ensure:
Full rules-as-written (RAW) fidelity when implemented
Protection of determinism, replayability, and event ordering
Prevention of accidental partial or premature implementations during earlier packets
This document exists to ensure grapple is not misinterpreted as a “simple combat maneuver” and is treated as the stateful combat system it is under RAW.
## 2. RAW Characterization of Grapple
Under RAW, grapple is a multi-phase, multi-round state machine, not a single action.
A complete grapple sequence includes, but is not limited to:
Melee touch attack to initiate grapple
Opposed grapple check to establish grapple
Application of the grappled condition to involved creatures
Per-round grapple options resolved via opposed checks, including:
Damage
Pin
Escape
Movement
Weapon usage
Restricted spellcasting
Sub-state transitions (pinned, escaped) that change legal actions mid-combat
Size, strength, BAB, and special ability modifiers that affect checks
Multi-attacker grapple rules that alter check resolution
RAW grapple therefore: - Persists across rounds - Redefines the action economy for affected actors - Introduces conditional legality and sub-state transitions
## 3. Determinism Risk Assessment
The grapple system introduces several determinism-critical properties:
State explosion: bilateral and multi-actor state with nested sub-states
Action replacement: normal actions are replaced by grapple-specific options
Mid-turn invalidation: escape or pin resolution can invalidate later action legality
Implicit interrupt semantics: state transitions can occur mid-round
These properties require a mature, explicit turn controller, condition system, and event-ordering contract to be implemented correctly.
## 4. Project Policy: Phased Implementation (No Deviation from RAW)
### 4.1 Deferral Policy
Grapple mechanics MUST NOT be implemented prior to CP-18: Combat Maneuvers.
No packet before CP-18 may:
Resolve grapple checks
Apply grappled or pinned conditions as a result of player or monster actions
Introduce grapple-specific action options
This is a temporal deferral, not a rule change.
### 4.2 Required Substrate Before Implementation
Grapple may only be implemented after the following packets are complete and frozen:
CP-14: Initiative & Action Economy
CP-15: Attacks of Opportunity
CP-16: Conditions & Status Effects
CP-17: Saving Throws & Defensive Mechanics
These packets provide the minimum stable substrate required for RAW-correct grapple behavior.
## 5. Implementation Requirements (When CP-18 Begins)
When grapple is implemented, the following constraints are mandatory:
Dedicated state machine
Grapple must be implemented as an explicit state machine with defined transitions.
Explicit event model
Events must include (non-exhaustive):
grapple_initiated
grapple_established
grapple_check_resolved
pinned_applied
grapple_escaped
Condition-driven legality
Grappled and pinned states must restrict actions via the condition system, not ad hoc checks.
Turn-boundary discipline
No implicit interrupts; all state transitions must be ordered and replayable.
Full RAW coverage
All options listed in PHB 3.5e (pp. 155–157) must be implemented or explicitly documented as future work.
Partial or “lite” grapple implementations are forbidden.
## 6. Prohibited Shortcuts
The following are explicitly disallowed:
Treating grapple as a standard attack variant
Applying grappled/pinned as narrative-only flags
Implementing only damage or escape options
Folding grapple logic into generic combat maneuver handlers
Any such implementation would constitute a deviation from RAW and violate determinism guarantees.
## 7. Documentation & Review Requirements
CP-18 must explicitly reference this amendment.
Any future audit touching combat maneuvers must confirm compliance with this document.
No PR may introduce grapple-related logic prior to CP-18.
## 8. Summary
This amendment ensures that grapple is:
Not forgotten
Not partially implemented
Not simplified
Not destabilizing to the combat engine
Grapple will be implemented fully, faithfully, and safely, or not at all.

## Phase 3.6 — Combat Engine Specification
### Purpose
Define combat as a deterministic, auditable execution contract layered on top of the core Rules Engine. This phase formalizes how combat state is tracked, how turns resolve, and how mathematical operators are applied in strict order, ensuring absolute correctness without exposing video‑game abstractions (e.g., HP bars) to players.
### 3.6.1 Combat as a System
Formal definition:
Combat = (WorldState₀, InitiativeOrder, TurnLoop, ResolutionOperators, EndConditions)
Where: - WorldState₀: Snapshot of all entities, terrain, effects, and hidden information at combat start. - InitiativeOrder: Ordered list of combatants, mutable only by ready/delay effects. - TurnLoop: Deterministic per‑round execution cycle. - ResolutionOperators: Pure mathematical functions (attack, save, opposed check, etc.). - EndConditions: Explicit triggers that terminate combat.
Combat is a bounded execution context with stricter time and action constraints than normal play.
### 3.6.2 Combat State Model
Combat state is partitioned into four mandatory layers.
#### A) Spatial State
Position (grid or abstracted distance)
Size category and occupied space
Reach and threatened squares
Movement modes (land, fly, swim, burrow, climb)
Terrain interaction (difficult terrain, elevation, squeezing)
Line of Sight (LoS) and Line of Effect (LoE)
Cover and concealment categories
#### B) Temporal State
Initiative order
Current round count
Start‑of‑turn and end‑of‑turn hooks
Duration tracking (rounds, minutes, hours)
#### C) Entity State (DM‑canonical)
HP (current, max, temporary) — exact values DM‑only
Armor Class breakdown (normal, touch, flat‑footed)
Saving throws (Fort, Ref, Will totals)
Attack routines and damage expressions
Conditions and status flags
Action economy usage (standard, move, full, swift, immediate, AoOs)
Resources (spell slots, charges, rage rounds, etc.)
#### D) Information State
Public vs private information boundaries
Hidden creatures, traps, and zones
Uncertainty gates (concealment, miss chance, SR)
### 3.6.3 Canonical Combat Algorithm
#### Combat Start
Establish initial positions.
Determine surprise and awareness per side.
Resolve surprise round (standard or move only).
Roll initiative for all combatants.
Lock initiative order.
#### Per Round Loop
Repeat until an EndCondition is met.
#### Per Turn (Strict Order)
Apply start‑of‑turn effects.
Determine legal actions (conditions, restrictions).
Select action package (standard/move/full/etc.).
Resolve movement (costs, legality, AoO triggers).
Resolve primary action (attack, spell, maneuver, skill).
Apply secondary gates (SR, concealment, crit confirmation, immunities).
Apply damage or effects.
Update state (HP, conditions, positions, resources).
Apply end‑of‑turn effects.
Decrement durations.
### 3.6.4 Resolution Ordering Contract
All combat effects must resolve in the following order: 1. Target legality (range, LoS, LoE, type). 2. Primary roll (attack, save, or opposed check). 3. Immunity checks. 4. Concealment or miss chance. 5. Spell Resistance (if applicable). 6. Damage or effect calculation. 7. Mitigation (DR, ER, hardness). 8. Reaction triggers (AoOs, readied actions, immediate effects). 9. State update and duration ticking.
### 3.6.5 Mathematical Operator Set
Core operators: - d20 + modifiers ≥ target - Damage dice + static bonuses - Multiplier application (crit rules) - Mitigation application (DR, ER, immunity) - Opposed checks - Miss chance checks - Spell Resistance gates
All combat math is expressed as compositions of these operators.
### 3.6.6 Action Economy Enforcement
The engine must explicitly enforce: - Action slot availability per turn - AoO limits (including Combat Reflexes) - Swift and immediate action constraints - Ready and delay initiative mutations - Five‑foot step restrictions
Illegal actions must be rejected before resolution.
### 3.6.7 Conditions as Rule Modifiers
Conditions are not labels; each condition is a bundle of rule atoms that: - Modify attack rolls, AC, saves, speed, actions, or visibility - Impose or remove action permissions - Define duration and removal triggers
Condition application and removal occur only through state updates.
### 3.6.8 HP Presentation Policy
Exact HP values are tracked internally only.
Player‑facing output uses descriptive health states (e.g., fresh, wounded, staggering).
No health bars or numeric HP are exposed to players.
### 3.6.9 End Conditions
Combat terminates when one or more of the following is met: - All hostile entities neutralized - Surrender or flight - Objective achieved - Total party defeat or escape
Post‑combat cleanup (stabilization, resource accounting) follows core rule procedures.
### Phase Output
A formally specified combat execution contract that integrates cleanly with the Rules Engine, supports voice‑driven DM adjudication, and guarantees mathematically correct outcomes under D&D 3.5 RAW.
## Phase 3.7 — DM Decision Policy Engine (Combat Choice Model)
### Purpose
Implement the non-rules part of combat DMing: how enemies (and encounter-level pacing systems) choose actions while remaining: - Rules-legal (all options validated by the Rules Engine) - Fiction-consistent (monster doctrine/personality) - Fun/fair (agency + telegraphing + guardrails)
This is a policy layer that sits above the deterministic Rules Engine. It never changes math; it only selects among legal actions and shapes presentation.
### 3.7.1 Inputs (Feature Set)
At each decision point, the policy receives a structured feature bundle:
#### A) Encounter-Level Features
Encounter purpose (tutorial/resource tax/boss/chase/puzzle-combat/narrative beat)
Win conditions (kill/delay/escape/protect/disrupt/hold)
Target rounds (expected length)
Reinforcement triggers/timers
Environment topology (terrain channels, hazards, visibility)
#### B) Side-Level Features
Action economy (effective actions/round after control)
Threat distribution (who is dealing damage/control)
Resource posture (observed slots/charges/HP bands)
Morale posture (disciplined/cowardly/fanatical/cunning)
#### C) Per-PC Inferred Features (Belief-Based)
Maintain belief distributions (not ground truth) over: - Defense profile (AC/touch/ff bands; save bands) - Vulnerability flags (low Will/Ref; immobility; lack of ranged) - Offense profile (DPR band; control potency) - Mobility profile (flight/teleport/escape tools) - Role signature (striker/controller/support/tank/scout)
#### D) Per-Monster Features
Doctrine / goal (protect leader, kill healer, delay, steal, escape)
Burst potential and recharge/limited-use tracking
Control suite (stun/grapple/fear/trip)
Survivability (AC/DR/ER/SR + escape tools)
Position leverage (reach/AoOs/flight)
#### E) Local Tactical Features
Distances + reachable squares
Threatened squares + AoO risk
LoS/LoE availability
Cover/concealment vs key attackers
Flank opportunities and choke points
Buff/debuff timers (remaining rounds)
### 3.7.2 Objectives (Utility Function)
Represent DM choice as optimizing a weighted utility:
U = w₁·Danger + w₂·Agency + w₃·Pace + w₄·RulesIntegrity + w₅·FictionIntegrity
Weights are set by encounter purpose and monster doctrine.
#### Quantifiable Proxies
Danger: PC drop probability (1–2 rounds), TPK risk (bounded), resource burn rate, action denial risk
Agency: counterplay availability, telegraphing strength, outcome sensitivity to player choices
Pace: decision density, swing moments frequency, resolution latency (avoid complex stalls)
### 3.7.3 Candidate Generation (Legal Action Enumerator)
Given the current state, generate only rules-legal candidates: - Damage line (focus fire / burst) - Control line (deny key actions, reposition via forced movement) - Reposition line (cover, break LoE, set flanks, choke control) - Objective line (interact/escape/trigger hazard) - Defensive line (withdraw, total defense, heal, buff) - Exit line (flee/surrender/bargain/hostage) when morale triggers
#### Authoritative Legality Checker Contract
The legality checker is the single source of truth for what is allowed.
The policy layer may suggest intent, but the legality checker must:
enumerate legal actions,
validate prerequisites (actions available, components, conditions),
validate targeting (range, LoS/LoE, AoE),
validate movement/path legality (terrain cost, AoO triggers),
reject illegal actions with explicit reasons.
The policy must never execute an action that did not pass legality validation.
If all high-scoring options are illegal, the policy re-scores the remaining legal set.
All candidates must be validated by the authoritative combat legality checker before scoring.
### 3.7.4 Scoring (Per-Action Evaluation)
For each candidate action a, compute: - P_success(a) using discrete d20 math (exact counts, not approximations) - EV_damage(a) or EV_effect(a) including mitigation and miss gates - ΔActionEconomy(a) (expected actions denied / preserved next round) - ΔPositionLeverage(a) (cover, flanks, AoO advantage, LoE denial) - ΔSurvival(a) (monster survival probability change) - Counterplay(a) (does this leave meaningful responses?) - Complexity(a) (penalize slow adjudication lines) - FictionFit(a) (doctrine/intelligence/morale constraints)
Example composite:
Score(a) = α·EV(a) + β·Leverage + γ·Survival − δ·Unfairness − ε·Complexity
#### Probability Notes (Engine Contract)
Attack rolls: nat 1/20 bounds apply
Saves: no auto fail/succeed in RAW; compute discretely
Crits: P_crit = P_threat × P_confirm (apply concealment contract consistently)
### 3.7.5 Selection Policy (Avoid Robot Predictability)
Choose actions via controlled stochastic selection: - Argmax when clearly dominant - Softmax over top-k when close (temperature/“spice” parameter) - Personality affects temperature and aggression/caution balance
This produces variety without breaking fairness.
### 3.7.6 Belief & Uncertainty Update Loop (Conservative Learning Contract)
The policy must operate under incomplete information: - Unknown HP (by design) - Unknown saves/DR/ER/SR initially
Maintain belief distributions and update them after observed outcomes, but update conservatively to prevent over-learning from dice variance.
#### Conservative Update Rules
Use evidence thresholds: do not shift a belief class after a single unusual result (e.g., one crit, one high/low roll).
Weight updates by likelihood under current belief (surprising outcomes update more, but still bounded).
Apply decay / smoothing so early observations do not permanently lock in.
Track sample size per hypothesis (e.g., fire resistance belief requires ≥2–3 relevant observations).
Treat crits, concealment misses, and failed confirmations as high-variance signals and down-weight them.
#### Example belief updates (bounded)
If fire spell repeatedly deals reduced damage across multiple casts → increase belief in fire resistance.
If Will saves repeatedly fail across multiple independent effects → lower Will band belief.
Beliefs influence future scoring but never override RAW legality.
### 3.7.7 Guardrails (Explicit Social Contract + Fairness Telemetry)
Hard and soft constraints that prevent “gotcha DM” behavior and prevent drift into optimal-but-unfun action denial.
#### Guardrail Categories
A) Hard constraints (must not violate) - No repeated hard-lock loops (stun/pin/fear chain) without counterplay unless encounter is explicitly flagged as apex. - No un-telegraphed lethal spikes as default policy (exceptions: table preference or explicit encounter tag). - Always preserve at least one viable response line when feasible (move, cover, dispel, retreat, break grapple, etc.).
B) Soft penalties (discourage but allow when justified) - Penalize action denial if it reduces player turns for multiple rounds. - Penalize focus fire if it produces early drops in non-apex fights. - Penalize high-adjudication-complexity lines if pacing budget is tight.
#### Fairness Telemetry (inputs the policy must track)
Early-drop risk: ≥2 PCs down within first N rounds.
Counterplay deficit: no viable responses detected.
